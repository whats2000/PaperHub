# Plan F6.1 — Slide Narrative Planning as an Orchestrator↔Worker Loop (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Make a multi-paper slide deck the way a PhD student would — understand the papers, choose a talk structure that fits the *type* of talk, plan a concept-driven backbone, go back into the papers to fetch the specific evidence each concept needs, and write a deck that tells the user's intended story as one connected argument (not isolated per-paper explanations).

**Architecture:** Replaces the first F6.1 one-shot build. A **deterministic `sl_seed`** builds a high-level map per paper (abstract + section list + figure inventory + survey-or-not) — the orchestrator's understanding *and* its dispatch menu. An **`sl_outline` orchestrator** selects a `narrative_pattern` archetype, plans a concept backbone against the slide budget, and **dispatches aimed `request_context` tasks** to a **task-driven `gather_context` worker** for the specific detail each concept needs, refining for ≤4 rounds until the outline is well-structured and every slide is grounded. The drafter (`slide_agent`, retained) receives **the outline + the gathered context** and renders it 1:1, surfacing the narrative visibly (agenda/section dividers, connective lead-ins, synthesis). The `DeckOutline` is persisted so the speaker-notes "talk-over" and the F6.2 Sources panel can consume it.

**Tech Stack:** Python 3.11 · Pydantic v2 · LangGraph · `litellm` via `LlmAdapter.structured` + tool-calling · `aiosqlite` · pytest/pytest-asyncio · `uv`.

**Authoritative spec:** SRS §III-5.3 "v2.33 — Plan F6" block (the orchestrator↔worker loop, the `narrative_pattern` roster, the grad-student north star, concept-driven principle, survey decomposition).

## Acceptance criteria (the three questions this plan must answer "yes" to)

These are verified at the **real-API gate (Task 9)**, not by unit tests — pytest proves mechanism, the live run proves these.

- **Q1 — Does it reflect actual PhD-student work?** The orchestrator must *understand → structure → dig for specific evidence → write → iterate*, not summarize. Evidence: the trace shows the seed map, a chosen `narrative_pattern`, a concept backbone, and **≥1 round of aimed `request_context` dispatches that fetched detail the seed didn't contain**, then refinement. A deck built from one shallow pass fails Q1.
- **Q2 — Professional-grade slides like a PhD student would make?** Concept-per-slide (one idea, developed with evidence — not a bullet dump or a highlight reel), an agenda + section structure that matches the archetype, figures used where they carry the point, a synthesis close. *(Visual/theme polish is F6.3; F6.1 owns the structure/content professionalism.)*
- **Q3 — Does the talk-over convey the user's intent as ONE connected story, not isolated per-paper explanations?** The deck reads as a single argument toward `audience_intent`, with real paper-to-paper bridges; and the **speaker-notes "talk-over", when generated, narrates that connected story** (because notes consume the persisted outline's `narrative_arc` + per-slide `transition_from_prev`). A deck that is "here's paper A, here's paper B, here's paper C" fails Q3.

If the gate shows any of these failing, the phase is not done (same discipline that caught the first build's 8-slide thinness).

### Task 4: `sl_outline` orchestrator loop (the heart)

**Files:** Rewrite `backend/src/paperhub/agents/sl_outline.py`; Rewrite `backend/src/paperhub/llm/prompts/slides_outline_v1.yaml`; Test `backend/tests/agents/test_sl_outline.py` (rewrite).

**Design (the loop):**
1. **Select `narrative_pattern`** from `seeds` + `task` + budget (the prompt encodes the archetype roster + the "1 paper→single_paper / survey→taxonomy / compare→comparison / …" inference; user wording overrides). Surveys: instruct decomposition into branches as threads.
2. **Round loop (≤ `MAX_ROUNDS=4`):** the LLM is given the seeds + the budget + the accumulated gathered context so far, and emits a **round action**: either
   - `dispatch`: a list of `ContextRequest{aim, paper_id}` (the specific evidence the concept backbone still needs), OR
   - `finalize`: the full `DeckOutlineDraft` (concept-per-slide backbone with `support_excerpts` filled from the gathered context).
   On `dispatch`, run the requests **in parallel** via the Task-3 worker (`run_gather_context(aim=…, paper_ids=[paper_id])`), append the returned excerpts + chunk IDs to the accumulated context keyed by aim, and loop. On `finalize` (or at round budget), resolve the draft → `DeckOutline` (assign `slide_index`; set each slide's `grounding_chunk_ids` from the chunk IDs of the context it cites; clamp `figure_key`/`paper_id` to the seed inventory — keep the Task-3-of-old hardening), and return `OutlineResult`.
3. **Termination:** `finalize` emitted, OR `MAX_ROUNDS` hit (then force a finalize call). Budget: `target_slides = parse_slide_budget(...)` is passed in and put in the prompt as the depth target.
4. **Trace:** one `report:outline` step recording, per round, the action, the dispatched aims, and the returned chunk IDs; the final result records the pattern + full outline + `rounds_used`.

  - the worker (a stub `gather_fn` injected into `run_sl_outline`) was called once per dispatched aim with the right `aim`/`paper_id`;
  - the returned `OutlineResult.rounds_used == 2`;
  - a finalized slide's `grounding_chunk_ids` equals the chunk IDs the stub worker returned for the aim that slide cited;

  - a forced-finalize path: a stub that never emits `finalize` still returns a valid outline at `MAX_ROUNDS` (assert `rounds_used == 4` and a non-empty outline).
  Inject the worker as a parameter — `run_sl_outline(..., gather_fn=<callable>)` defaulting to the real `run_gather_context` — so the loop is unit-testable without an LLM or DB-heavy gather.

## Self-review against the three questions (author, fresh eyes)

- **Q1 (PhD-student work):** Addressed structurally — the loop is *understand (seed) → structure (archetype) → dig for specific evidence (aimed dispatch) → write from material → iterate*. The plan makes the dig **observable and required** (Task 9 gates on ≥1 dispatch round + `rounds_used≥2`), so a degenerate one-shot can't silently pass. **Residual risk:** the *quality* of the digging is prompt-dependent and only provable at the live gate — unit tests (Task 4) prove the loop mechanism, not that the LLM digs wisely. Called out, not hidden.
- **Q2 (professional slides):** F6.1 owns **structural/content** professionalism (concept-per-slide from evidence, archetype-shaped agenda + sections + synthesis — Tasks 4, 5). **Honest gap:** *visual* professionalism (theme, color, typography) is **F6.3**, not this phase — so "looks professional like them" is only half-delivered by F6.1. Stated plainly so it isn't over-claimed.
- **Q3 (connected talk-over, not isolated):** Two delivery mechanisms, both in-plan: the **deck** reads connected (archetype + bridges rendered as visible lead-ins — Task 5), and the **talk-over (notes)** narrates the connection (notes consume the persisted `narrative_arc` + transitions — Tasks 6–7). # F6.1-R — Gather Rework: PaperDigest + Targeted Reads + Always-On Streaming

> **For agentic workers:** same execution rules as above (subagent-driven, TDD, bite-sized). This rework GATES the F6.1 merge — the narrative work above is validated but not shippable until the gather is reworked (SRS §III-5.3, the "heavy gather / no-stream" lesson + commit `75859b9`).

**Why:** the live UX gate showed the F4.5 `gather_context` worker is a *"summarize the whole paper"* agent mis-used as an incremental reader — a blind whole-paper read per paper + a full flagship read per dispatched aim (8 redundant flagship reads of one paper observed), frontend frozen for minutes. Fix = **compress once (cached digest) → structure from it → deterministic `read_section` for exact evidence only → stream throughout.** Removes the blind gather and the per-aim LLM gather agent entirely.

**Goal:** a deck whose context is gathered the way a person works — skim the gist of each part once (cached), then re-read only the exact detail a slide needs — fast, cheap, fully streamed.

**Tech stack:** Python 3.11 · Pydantic v2 · `aiosqlite` · `LlmAdapter.structured` on the **small** model (`report_section_model`) for compression · pytest · `uv`.

**Conventions:** backend cmds from `backend/` via `uv`; strict ruff + mypy; TDD; branch `feat/slide-narrative-planning`; full gate + real-API gate at the end. **Do NOT edit `src` while a live run is in flight** (`--reload` restarts orphan runs).
