# PaperHub F4.4 — Slide Quality Refinement (Agile Calibration Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to execute this plan ROUND-by-ROUND. Steps use checkbox (`- [ ]`) syntax for tracking. **This plan is intentionally agile-shaped — the round bodies will evolve from each comparison cycle, not pre-committed.**

**This plan is a PROCESS description, not an implementation spec.** It tells the worker WHAT to study, WHAT to compare against, and HOW to iterate. It deliberately does NOT prescribe the prompt rewrites or node-topology changes — those emerge from doing the study. Treat any "predicted change axis" below as a *forecast to be tested*, not a commitment.

**The CORE goal is to design a generally better slide pipeline — NOT to reproduce the reference decks.** The references exist so the *developer* learns what professional output looks like; the pipeline's job is to be capable of that quality on ANY paper and ANY paper count, not just the 3 reference papers. Production testing across varied paper counts and topics is done by the user manually, against the user's own paper choices — not by this plan's harness.

**HARD CONSTRAINT — do not leak the comparison TARGET into LLM prompts.** Two categories of reference material, two different rules:

- **Comparison targets (FORBIDDEN as runtime LLM context).** The dev-set gold artifacts we're SCORING AGAINST: `D:\GitHub\Final_Report\slides.tex`, `D:\GitHub\Final_Report\<arxiv_id>\slides.tex`, `D:\GitHub\Final_Report\speaker_notes*.tex`, and the generated deck PDFs. These MUST NOT be embedded — verbatim or summarised — into any runtime LLM prompt, few-shot example, or retrieval-augmented context. If the LLM sees the target, it reproduces the target instead of applying the principles, and the dev-set scores become meaningless on novel papers (the user's production cases). The harness COMPARES PaperHub output to these artifacts; the pipeline NEVER reads them.

- **Methodology references (ALLOWED to inform prompt design).** `reference/paper2slides-plus/prompts/config.yaml`, the `.skill` SKILL.mds + their references (`latex-pitfalls.md`, `word-budgets.md`, `speakable-vocabulary.md`) — these are design documents *about how to build talks well*, not specific artifacts being scored against. The developer can study them, paraphrase their rules, port their structural patterns, even adopt their template structures (paper2slides-plus's prompt templates are themselves design references for ours). It's fine for our new prompts to encode principles like *"figure frames carry only a figure + a 1-sentence caption"*, to include a GOOD-vs-BAD example pair authored by us, to ship a layout-pattern template the LLM fills in for a `concept_2col` slide. That's the prompt doing its job.

The line: **prompts encode generalizable PRINCIPLES + DESIGN PATTERNS authored by the developer (even substantial, even template-shaped); prompts NEVER ship the specific dev-set artifacts the harness scores against.** When in doubt: ask "if I score this round against a completely new paper the LLM has never seen, would my prompt design still produce good output?" If the design depends on the LLM having seen the dev-set artifacts at runtime, the design is leaking.

**Goal:** Make a PaperHub-generated deck for the *same three papers* used in `D:\GitHub\Final_Report` (a hand-built gold-standard talk) look and read like a professional conference deck — at least at parity with that reference — when the user just says *"make me slides for these three papers"*. Today the new F3/F4 pipeline produces decks users describe as "terrible", "unprofessional", "not laid out like before". User reports specifically say **the older `reference/paper2slides-plus` project was much better** — even on single-paper decks the new F3/F4 fragmented pipeline regressed against it. So there are **TWO target outcomes to constraint-check against**:

1. **`reference/paper2slides-plus` (single-paper, beloved)** — the old project's prompts encode taste densely: layout patterns, density rules, the equation+itemize explanation pattern, the *only-figure-and-caption-in-a-figure-frame* rule, "no Thank-you slide", a specific Berlin/dolphin/professionalfonts/14pt preamble with `\input{ADDITIONAL.tex}` to pull the paper's own custom commands. Single LLM call sees the whole paper + whole figure list + whole instruction set and emits the whole `slides.tex` at once. The contract is in one place, the LLM is trusted, the output is coherent. **Single-paper PaperHub output today must be AT LEAST AS GOOD as paper2slides-plus on the same paper.**
2. **`D:\GitHub\Final_Report` (multi-paper conference deck, hand-built gold)** — the multi-paper extension: same density discipline, plus connective tissue (a *"Three Bottlenecks"* table that frames the 3 papers as 3 axes of one problem), per-paper concept+math PAIRS (each paper gets exactly 2 slides), an original *"Proposed Direction"* TikZ synthesis, a take-away + open-question + Thank-you closer, 3-column references-with-QR slide, per-language speaker notes that fit a real word-budget by slide weight and read aloud cleanly. **Multi-paper PaperHub output must reach this standard.**

**Why agile-shaped:** the gap is *quality*, not a missing feature. We don't yet know whether the fix is one prompt rewrite, three node-prompt rewrites, one new deterministic stage, a new audit node, restoring monolithic-context calls, or all of the above. Pre-committing a task list would either over-shoot (forcing changes we don't need) or under-shoot (missing the actual lever). The plan instead defines a **calibration harness** (same 3 papers → diff vs paper2slides-plus on each individually + diff vs Final_Report as a whole → score → pick the next change) and a **prediction** of likely change axes, then iterates rounds until the comparison passes a per-dimension threshold.

**Two parallel goals (the user's framing):**
1. **The gold-standard methodology becomes the DEFAULT profile** — every PaperHub user gets a conference-quality deck from a bare `"make slides"` ask, with no extra prompting.
2. **Style is per-user configurable** — theme, audience level, closer style, structural depth (5-min vs 10-min), language preferences live in a `SlideStyleProfile` that defaults to the gold profile but can be overridden via the chat (deck-command classifier already routes "use the metropolis theme / casual tone / shorter / no QR codes" turns to the editing path) or via active memories (e.g. *"always present in Traditional Chinese with a take-away closer"*).

**Architecture (no new core deps):** F4.4 is mostly *prompt + topology refinement* of the existing `report_graph.py` subgraph plus one optional new deterministic node (`sl_audit_claims`) and one new YAML config surface (`slide_style_profile.yaml`-like). It does not change F2 (Marker → `PaperAsset`), F3's compile/verify-figures contracts, F4's decoupled NOTES/EDIT flow, or F4.2's title/preamble editing. It builds on top of all of them.

**Tech Stack:** Python 3.11, the existing `slides_narrate_v1` / `slides_draft_frame_v1` / `slides_note_author_v1` / `slides_coherence_v1` / `slides_understand_v1` YAML prompt registry, the existing `sl_*` LangGraph nodes in `backend/src/paperhub/agents/report_graph.py`. New: a per-user style profile (dataclass + a small REST surface, or piggyback on active memories). Optional new node: `sl_audit_claims` (deterministic + LLM mix). No new DB schema except a `slide_style_profile_json` column on `chat_sessions` (or a memory-only path — TBD in Round 1).

### A. `reference/paper2slides-plus/prompts/config.yaml` — the OLD, single-paper, beloved pipeline

This is the "people said this version is much worse" baseline. Read the full file (~390 lines) and understand each stage's contract. The salient design choices to distill (and the worker must verify these by reading the file, not by trusting this summary):

- **`stages.initial`** — Single LLM call, given `{latex_source}` (the WHOLE paper LaTeX), `{figure_paths}` (the whole inventory), `{audience}`, `{num_slides}`, `{max_items}`, and the full Berlin/dolphin preamble template baked into the prompt. The LLM emits the COMPLETE `slides.tex` file at once. The prompt is ~80 lines of prescriptive rules: "Do not say *'better accuracy'* — say *'14% better accuracy on benchmark X'*"; the **equation+itemize pattern** (`\begin{align}` then `\begin{itemize}` explaining notations); the **figure-only-frame** rule ("DON'T use any text other than the caption of the figure"); explicit "Do NOT include a 'Thank you' slide"; specific abbreviations (GANs/VAEs/SGD/LLM, w/, w/o); the preamble must include `\input{ADDITIONAL.tex}` to pull the paper's own `\newcommand`s. Audience is calibrated up front ("ML professors, PhD students, AI research scientists").
- **`stages.update`** — Read paper + current beamer, **enforce ≤15 words per `\item`** with the explicit instruction *"If a slide has too much text overall or items are still too long after rewriting, split it into multiple frames"* — the size-overflow strategy is **split, not shrink-font**. Strip `\cite`/`\bibliography`. Re-check figure paths exactly.
- **`stages.revise`** — Read beamer + **ChkTeX linter output**, fix syntax/escapes/figure-paths. This is a *deterministic-linter-feedback* loop the OLD project had that PaperHub today does NOT — PaperHub's `sl_revise` only reacts to Overfull warnings from `pdflatex`, never to ChkTeX warnings.
- **`stages.interactive_edit_single_slide`** — Edit ONE frame, instruction explicitly says *"If the instruction asks to split the frame into multiple slides/frames, you MUST return ALL the new frames"*. PaperHub's F4 `edit_frame` prompt *forbids splitting* (the F4 plan even calls out the edge case where a split would drop notes). The OLD project decided splits ARE part of an edit — and used it as the primary overflow mitigation.
- **`stages.generate_speaker_notes`** — Single LLM call given the WHOLE paper + WHOLE beamer code, emits notes for ALL slides at once with `[SLIDE N]` markers. Has explicit **GOOD vs BAD example** in the prompt (descriptive *"This slide discusses state variables..."* → conversational *"So now let me talk about state variables. What's interesting here is..."*). 2–4 sentences per slide. First person, conversational.
- **`stages.edit_speaker_notes`** — Revise prior notes when the deck has changed; matches notes to slides by CONTENT not by index; emits notes for ALL `{frame_count}` slides; drops orphan notes; writes fresh notes in matching style for new slides.
- **`stages.interactive_qa`** — Q&A about the paper WHILE viewing the deck; explicit rule: "*the paper is AUTHORITATIVE; the slides are CONTEXT for what the user is looking at*". Plain Markdown, no LaTeX block.

The salient OLD-vs-NEW architecture deltas the worker must document in the Round 0 gap analysis:

| Dimension | paper2slides-plus (OLD) | PaperHub F3/F4 (NEW) | Suspected impact |
|---|---|---|---|
| **Per-stage context width** | Full paper + full figure inventory + full instruction set in ONE LLM call | Fragmented: understand sees one paper alone; narrate sees briefs only (no chunks); draft sees one frame's chunks only; coherence sees frames but no paper context | Contract knowledge lives in the prompts, but no single call sees enough state to act on the full contract |
| **Contract density** | One ~80-line prompt encodes all layout / density / figure / equation / abbreviation rules | Spread thinly across 5 prompts; many rules dropped or weakened | Drift on every rule the OLD prompt was specific about |
| **Overflow strategy** | "Split the frame" (explicit instruction in `update`) | "Revise to shrink" (`sl_revise` reacts to Overfull) — no split path | F3 frames silently overflow when shrinking can't help |
| **Linter loop** | ChkTeX warnings → revise | pdflatex Overfull → revise; no ChkTeX | Escape-char errors, `73%` comment-out bugs, ampersand mistakes survive |
| **Speaker notes** | Single call, whole-deck context, GOOD/BAD examples in prompt | Per-slide call, one frame's TeX only, no examples | Notes lose cross-slide connectors ("Remember from slide 3…"); lose conversational pacing |
| **Edit-splits-frame** | Explicit "if splitting return ALL new frames" | Edit_frame forbids splitting | Lost overflow recovery path via user edit |
| **Audience calibration** | Up-front audience string in every prompt | Implicit — no audience parameter | Output is generically academic, can't tune for cross-domain Master's audience |
| **`ADDITIONAL.tex`** | Preamble includes paper's own custom commands | F2 `PaperAsset` doesn't preserve them | Math that depends on the paper's `\newcommand` shortcuts breaks silently |
| **Figure-frame rule** | "DON'T use any text other than the caption" — figure frames are ONE figure + ONE caption, nothing else | F3 draft allows figure + bullets in the same frame | Figure slides cluttered, lose visual impact |
| **Equation+itemize pattern** | Explicit `\begin{align}` + `\begin{itemize}` notation-explanation pattern | No pattern; equation slides have bullets above OR below ad hoc | Math slides feel scattered |

This is the *forecast*; the Round 0 deliverable confirms or revises it.

### C. The reform challenge — scaling OLD contract clarity to multi-paper without context blow-up

paper2slides-plus's monolithic-context approach is BELOVED on single-paper but is **not directly portable to multi-paper**: 3 full-paper LaTeX sources + their figure inventories + their custom commands would exhaust the LLM context window for any reasonable paper length. So the reform cannot be "just put back the OLD monolithic prompt"; it has to find a topology that **preserves OLD contract clarity** while staying within context budget for N papers.

Candidate topologies for the worker to evaluate in Round 1 (forecast — pick by Round-0 evidence):

- **(i) Per-paper sub-deck then merge.** Run paper2slides-plus-style monolithic call PER paper → get a "focused single-paper deck" per paper. Then a final pass takes those + the cross-paper brief and INTERLEAVES + adds connective tissue (bottlenecks-overview, proposed-direction, take-away closer). Concretely: replace `sl_understand` + `sl_narrate` + `sl_draft` with `sl_per_paper_deck` (fan-out, one OLD-style monolithic call per paper) + a new `sl_merge_multi_paper` stage. Coherence + assemble + verify + compile + notes stay. This keeps OLD contract clarity per paper.
- **(ii) Plan-then-render.** First call: plan the cross-paper deck (names the patterns: bottlenecks-overview, concept+math pairs, proposed-direction). Second call: monolithic render given the plan + paper BRIEFS (compressed, not full LaTeX). This is closer to the current topology but restores monolithic-context for the actual rendering call.
- **(iii) Fragmented topology + restored context.** Keep today's nodes, but each node's prompt gets the FULL outline + ALL paper briefs + ALL figure inventories (not just the assigned slice). Smallest topology change, biggest prompt rewrite. The "context width" delta from the OLD-vs-NEW table is the main thing to restore.
- **(iv) Hybrid.** OLD-style monolithic per-paper for the per-paper concept+math pairs; orchestrator-style for the bridging slides (motivation, bottlenecks, proposed-direction, closer).

The Round 0 deliverable picks ONE candidate as the Round 1 hypothesis. Each round measures: did this candidate close the OLD-vs-NEW contract gap?

## The gold standard, distilled

The reference at `D:\GitHub\Final_Report` ships with three explicit skill docs (`academic-talk-deck.skill`, `talk-speaker-notes.skill`, `paper-claim-audit.skill` — each a ZIP of a `SKILL.md` + references + scripts; extracted to `_skill_extract/`). These are the contracts we are calibrating *against*. Summary:

### 1. `academic-talk-deck` — the structural skeleton

A conference talk follows a *named pattern*, not a generic "intro/method/results/conclusion":

| # | Slide | Pattern |
|---|---|---|
| 1 | **Title** | `[plain]` frame with `\titlepage`; institute + venue ID; one-liner author list of all papers covered. |
| 2 | **Papers Covered** | 3-column layout, each column an `[s]`-stretch minipage with: paper [N] tag, *italic short title*, tiny author list, venue badge (e.g. **ICLR 2026** in accent color), `\vfill`, QR code, URL underneath. |
| 3 | **Motivation / The Problem** | One large figure (`0.65\linewidth`, `keepaspectratio`), one bolded sentence beneath. |
| 4 | **Bottlenecks Overview** | `booktabs` 3-row table mapping (problem → paper → speedup), wrapped by a `\begin{block}{tagline}`. This is the *connective tissue* — explicitly frames the three papers as three axes of one problem. |
| 5–10 | **Per paper × 2 slides** | (a) **Concept** — 2-column with figure on left (~0.55–0.60 width), 2–4 ultra-short bullets right, `\begin{block}{Result}` at the bottom with the headline number. (b) **Math** — vertical stacking with `\textbf{Section heading:}` above each `\[...\]`, max 2 equations, an optional one-line `\textcolor{accent2}{\small ...}` annotation under the last. |
| 11 | **Proposed Direction** | The speaker's own synthesis: usually a TikZ diagram (scheduler → axes → unified policy) on the right, the central function-style equation + bullets on the left. |
| 12 | **Plan** | Numbered list, 4–6 steps. |
| 13 | **Take-away + Open Question** | `[plain]` frame, two `\rule` horizontal lines framing one take-away sentence, `\begin{block}{Open Question}` with question in italics, close with `\textbf{\Large Thank you. Questions?}`. |

**Non-negotiables that wreck decks if skipped:**
- **Use the original paper's figures verbatim** (PaperHub's F3 `sl_verify_figures` already enforces this — keep it).
- **Pull math from `ORIGINAL_PAPER.tex`, don't paraphrase.** Compress notation to fit one line is OK; invented symbols are not. **Today's `sl_narrate` lets the LLM write equations as free LaTeX — no provenance check.** This is a gap.
- **Each slide is one idea.** If you reach for bullets, you have two slides masquerading as one. Today's `sl_draft` allows ≤4 bullets — fine — but the *narrate* stage routinely produces 5–7-bullet slides because the per-slide-budget concept doesn't exist in the prompt.
- **Spoken detail lives in notes, not on the slide.** Today's `sl_draft` says this; today's `sl_narrate` undermines it by emitting `key_points` lists that the draft then dutifully renders.

### 2. `talk-speaker-notes` — the spoken-time budget

Notes match a real spoken duration, by *slide-weight-aware* per-slide word budgets:

- **Pacing**: English 150 wpm, Traditional Chinese 200 字/分, Japanese 300 字/分; multiply by `target_duration × 0.9` (10% slack for pauses).
- **Slide weights**: content=2, detail/math=1.5, transition (title/refs/dividers/closer)=1. Per-slide budget = `total × weight / sum(weights)`.
- **2–4 sentences per slide max** (current prompt says 3–6 — drift target).
- **First-person, why-not-what**: the slide already shows the *what*; the voice carries the *why*.
- **No em-dashes (`--`, `—`) in the spoken body** — they trip the speaker. Replace with conjunctions ("which is", "because", "so", "but").
- **Vocabulary calibration**: swap formal connectors (`orthogonal → separate`, `complementary → cover each other`, `however → but`, `crucially → and the key is`); keep load-bearing technical terms (VLA, denoising, autoregressive, codebook). For Chinese: `因此 → 所以`, `然而 → 但 / 不過`, `互補 → 彼此補位`, etc.
- **Important vs transition classification surfaces in the notes header** so the speaker knows where to lean and where to glide.
- **Filler cull**: drop "essentially", "actually", "really", "obviously", "in order to → to".

### 3. `paper-claim-audit` — the safety net

After the deck is built, audit every numerical claim, equation, and figure interpretation against the source paper. Catches:
- **Wrong numbers** (speedups / success rates / latencies that don't match the paper).
- **Conflated metrics** — the canonical example: a deck once said `NFE: 10 → 3.2 (6.5×)` but the paper said `6.5× inference time, 3.1× NFE`. The `6.5×` belonged on a different line.
- **Equation subscript/superscript drift, wrong sum bounds, wrong conditioning order in probabilities.**
- **Wrong figure interpretation** — slide caption claims "X vs Y" when the figure is actually "Ours vs Static across five phases".

This pass does **not** exist in PaperHub today. The closest is `sl_verify_figures` which only checks that figure *keys* are real — it does not audit *what the slide says about the figure*.

## The current PaperHub pipeline (what we're refining)

`backend/src/paperhub/agents/report_graph.py`, ~1100 LOC. The relevant chain when GENERATE fires:

```
sl_resolve → sl_understand (per paper, fan-out → PaperBrief)
          → sl_narrate     (one TalkOutline, all papers)
          → sl_draft       (per frame, fan-out → FrameDraft)
          → sl_coherence   (whole-deck rewrite)
          → sl_assemble    (preamble + figure inventory + concat)
          → sl_verify_figures (deterministic; replaces unknown keys with [figure omitted])
          → sl_compile     (pdflatex; up to 3 retries via sl_revise on Overfull)
          → sl_notes_finalize (one note per PDF page)
          → sl_emit        (deck SSE)
```

The prompt registry has separate YAMLs per node: `slides_understand_v1`, `slides_narrate_v1`, `slides_draft_frame_v1`, `slides_coherence_v1`, `slides_note_author_v1`, `slides_revise_v1`. (Plus the F4 edit/title/preamble/deck-command/target-language prompts, which F4.4 does not touch directly.)

**What today's prompts get wrong vs the gold standard** (the observed-not-assumed list — read the YAMLs side by side with the gold deck to confirm):

1. **`slides_narrate_v1` is generic** — it asks for `title → exec-summary → intro → method[bulk] → results → conclusion`. It does *not* know about the *named patterns* — references-with-QR slide, bottlenecks-overview table, concept+math pairs per paper, proposed-direction synthesis, take-away + open-question + thank-you closer. So it doesn't *plan* a slide of those shapes, and downstream `sl_draft` can't render what `sl_narrate` didn't ask for.
2. **`slides_draft_frame_v1` doesn't know the layout patterns** — no awareness of `\begin{block}{Result}`, 2-column `0.55/0.45` split for concept slides, vertical-stack math slides, the `[plain]` frame for the title and closer, the `\rule` + block pattern for the take-away.
3. **`slides_understand_v1` extracts a clean brief** but treats every paper symmetrically — no "this paper is the bottleneck axis #2 in a 3-paper talk" context. So the brief is the same whether the paper is being presented alone or as one of three.
4. **`slides_note_author_v1` says "3–6 sentences"** — drift from the speakable target of 2–4. It says "first person, explain WHY" — present and good. It does **not** know about per-slide weight budgets, em-dash sweep, vocabulary swaps, or filler cull.
5. **No claim-audit node exists** — `sl_verify_figures` only checks figure-key existence.
6. **No style profile** — theme is hardcoded in the preamble emitter; depth/closer/audience-level are implicit in prompt text.

## The calibration harness (developer scoring tool, NOT production benchmark)

The harness is how the *developer* measures whether a round improved quality. It is NOT a production benchmark, NOT a CI gate, NOT a contract the output must match the gold deck verbatim. The 3 reference papers are a fixed *development scoring set* so rounds are comparable to each other; production quality is judged by the USER on the user's own varied paper choices (different counts: 1, 2, 3, 5+; different topics; different languages). The harness exists because without a fixed scoring set, "did this round improve things" is unanswerable. Once F4.4 is done, the harness's job is finished — it's not a long-lived regression suite.

The 3 papers in `D:\GitHub\Final_Report` are arXiv (2509.22093v1, 2512.04952v2, 2602.20200v2). All are ingestible by PaperHub's existing arXiv path.

**Scenario A — Single-paper regression (run this for EACH of the 3 papers):** add ONE paper to a fresh session via `POST /papers`, then drive `POST /chat` with:
> `"Please prepare ~15 Beamer slides covering this paper for an ML research audience."`

This is the paper2slides-plus baseline scenario (single paper, 15 slides, ML audience). PaperHub output is compared against `D:\GitHub\Final_Report\<arxiv_id>\slides.pdf` (the hand-built per-paper deck) AND, where feasible, against a freshly-run `reference/paper2slides-plus` invocation on the same paper. ### H2 — The diff-vs-gold scorecard (two parts, run BOTH every round)

A markdown checklist scored 0/1 per dimension (LLM-as-judge optional, but a human eyeball pass is the gate). The dimensions ARE the contract clauses from BOTH baselines — Scenario A is scored against paper2slides-plus's contracts, Scenario B against the Final_Report gold + paper2slides-plus's universal rules. Score what was missed:

**Scorecard A — single-paper contract fidelity vs `reference/paper2slides-plus` (one card PER paper):**

| Dimension | Pass criterion |
|---|---|
| **Specificity** | No abstract claims; every metric quantified ("14% better", not "better"); benchmarks named |
| **Equation+itemize pattern** | When an equation appears, an `\begin{itemize}` follows it explaining notations + meaning. Not bullets-then-equation, not equation-alone |

| **Preamble carries paper's commands** | `\input{ADDITIONAL.tex}` or equivalent — paper-specific `\newcommand`s survive into the deck |
| **No Thank-you slide alone** | The deck does NOT end on a bare "Thank you" frame (paper2slides-plus's explicit rule) |
| **ChkTeX-clean** | The compiled deck passes ChkTeX with zero high-severity warnings on the slide source (modulo `ADDITIONAL.tex`-related noise) |
| **Abbreviation convention** | Standard abbreviations used (GANs/VAEs/SGD/LLM); `w/`, `w/o` for with/without; no em-dash (`—`) — use `---` |
| **Audience calibration** | Vocabulary matches the audience parameter (default: ML professors / PhD students); not generically academic |
| **Speaker-notes conversational** | First-person, 2–4 sentences, with cross-slide connectors ("Remember from slide 3…") |

**Scorecard B — multi-paper conference deck vs `D:\GitHub\Final_Report`:**

| Dimension | Pass criterion |
|---|---|
| **Skeleton order** | Title → Refs-with-QR → Motivation → Bottlenecks-overview → (Concept+Math)×N → Proposed-Direction → Plan → Take-away+Open-Question; Closer mirrors gold |
| **References slide** | 3-column layout, per-column: tag + italic title + tiny authors + venue badge + QR + URL. Stretched to equal height via `[s]` minipage |
| **Bottlenecks-overview** | A connective-tissue table or 3-row visual that maps each paper to one axis of the shared problem — not "3 separate paper sections" |
| **Concept slide layout** | 2-column figure-left ~0.55, bullets-right ≤4 short items, `\begin{block}{Result}` with headline number |
| **Math slide layout** | Vertical stacking with `\textbf{heading:}` above each `\[...\]`, ≤2 equations, no bullets |
| **Proposed-direction slide** | A *synthesis* not a summary — TikZ or table that says something about the three together. (Placeholder OK if PaperHub correctly identifies it can't synthesise.) |
| **Closer** | Take-away sentence framed by `\rule`s + `\begin{block}{Open Question}` italic + Thank-you/Questions. NOT "Thank you for listening" alone |
| **Figure fidelity** | Every `\includegraphics` resolves to a real file from the source paper (F3 already enforces — keep verified) |
| **Math fidelity** | Every equation traceable to a `\begin{equation}` in the corresponding `ORIGINAL_PAPER.tex`; no invented symbols (NEW for F4.4 — the audit pass) |
| **Per-language notes pacing** | Total words within ±15% of `duration × wpm × 0.9`; content slides 2× weight, transition slides 1× weight, detail 1.5× |
| **Spoken-prose (notes)** | No em-dashes in body; ≤4 sentences/slide; first-person; vocabulary calibrated to audience (no `orthogonal`, `complementary`, `crucially` in body); cross-slide connectors present |
| **Claim audit** | Every numerical claim in the deck appears verbatim in some source-paper chunk; no conflated ratios |

For each round, generate decks for BOTH scenarios (4 PaperHub decks per round: 3 single-paper + 1 multi-paper), fill in scorecards A×3 + B×1, pick the lowest-scoring dimension(s), and design the next round to attack those. A round that improves B but regresses A on a previously-passing dimension is a regression and must be re-designed.

### H3 — The harness scripts (Round 0 deliverable)

Committed scripts under `backend/benchmark/slide_calibration/`:

- `seed.py` — Two modes: `seed --single <arxiv_id>` adds ONE paper to a fresh session (Scenario A); `seed --multi` adds all 3 (Scenario B). Returns `(session_id, paper_ids)`.
- `run_single.py <session_id>` — POSTs the Scenario A prompt, polls until `deck` SSE, dumps `slides.{tex,pdf}` + `speaker_notes_*.tex` + `tool_calls.json` to `results/single/<arxiv_id>/round_<N>/`.
- `run_multi.py <session_id>` — Same for Scenario B → `results/multi/round_<N>/{en,zh}/`.
- `compare.py --round <N>` — Opens PaperHub round-N PDF side-by-side with the corresponding gold (`D:\GitHub\Final_Report\<arxiv_id>\slides.pdf` for single, `D:\GitHub\Final_Report\slides.pdf` for multi); generates a scorecard markdown stub (A×3 + B×1) the human fills in. Also runs the deterministic checks (em-dash sweep, word-count, figure-existence) so the human only judges what humans must judge.
- Optional: `run_paper2slides_plus.py <arxiv_id>` — invokes the OLD `reference/paper2slides-plus/api.py` on the same arXiv source to produce the *live* OLD baseline alongside the hand-built one. If running the OLD project is non-trivial in this env, fall back to the hand-built per-paper `<arxiv_id>/slides.pdf` for the comparison.

The 3 reference papers' per-paper `slides.tex` under `D:\GitHub\Final_Report\<arxiv_id>\` are the single-paper golds; the top-level `D:\GitHub\Final_Report\slides.tex` is the multi-paper gold; the three `.skill` ZIPs in the same folder are the methodology references.

## The agile loop (how rounds work — not what each round is)

Each round is one iteration of: **observe → pick one lever → change → re-run → score → decide**. Round N body is not in this plan; it is decided at the start of Round N from the Round N-1 scorecard. The shape:

```
Round N start:
  1. Read last round's scorecard. Pick the lowest-scoring dimension(s).
  2. Hypothesise the lever: a prompt rewrite (which YAML), a topology change
     (new node / merged node / re-ordered call), or a deterministic stage
     (e.g. claim-audit, vocab-pass post-processor).
  3. Apply ONE lever per round. Commit on its own branch task.
  4. Re-run the harness on the same 3 papers, same prompt, same language.
  5. Fill in the new scorecard. Diff vs last round (improved? regressed?
     unchanged?). If regressed on a previously-passing dimension, revert
     and design differently.
  6. Decide Round N+1: another round, or DONE if scorecard meets the
     done-definition (§Done).
```

Rounds are *not* time-boxed — they're scope-boxed. A round that needs 4 sub-commits to make its one lever land is still one round.

**Why one lever per round:** changing two things at once and seeing the scorecard change makes attribution impossible. The harness reproducibility (same 3 papers, same prompt, deterministic seed where possible) is wasted if the diff is multi-causal.

**Likely Round 5 (claim audit).** Add a new `sl_audit_claims` node between `sl_compile` and `sl_emit`. Conflated-ratio detection is the LLM part; equation-string-diff is deterministic. This is from `paper-claim-audit.skill` and does NOT exist in either baseline pipeline (paper2slides-plus's `revise` only handles syntax, not claim fidelity) — F4.4 introduces it.

**Likely Round 6 (style profile).** Introduce `SlideStyleProfile` dataclass + `slide_style_profiles.yaml` shipping a `default` profile (= the gold methodology: Berlin/dolphin/professionalfonts/14pt/16:9, take-away+open-question closer, references-with-QR, audience="ML PhD students"), plus 2–3 alternates (`casual_talk`, `defense_long`, `lightning_5min`, `metropolis_minimal`). Active memories carry persistent preferences (e.g. *"always present in Traditional Chinese with metropolis theme"*) — the F4.2 `_recompile_and_emit` helper already knows how to re-emit with new preamble; profile changes route through the same path. Audience is a profile field — paper2slides-plus's `{audience}` parameter, generalized.

**Possible Round 7 (proposed-direction = speaker's contribution placeholder).** For multi-paper talks, the gold deck's slide 11 is *original synthesis*, not a summary. PaperHub today has no concept of "the user's own contribution" — and that's fine for an academic-talk-deck default (we cannot fabricate a research direction). The right answer is probably to **emit a "Proposed Direction" placeholder slide** when ≥2 papers are in scope, with a prompt to the user *"add your own contribution here — what do these three together suggest?"* + a default rendered placeholder TikZ. Don't try to LLM-synthesise a research direction; that's hallucination territory.

**Possible Round 8 (`ADDITIONAL.tex` preservation).** paper2slides-plus's `\input{ADDITIONAL.tex}` flow preserves the paper's custom commands; F2's `PaperAsset` does not. Math that uses `\newcommand`-shortcuts from the source paper breaks silently. F4.4 should plumb the paper's custom-command block through the asset and into the emitted preamble (per paper, or merged with collision-rename when N>1). This is a non-trivial change but the OLD project considered it load-bearing.

**Will NOT be needed (forecast):** F2 Marker ingestion changes (PaperAsset is fine; `ADDITIONAL.tex` plumbing is a small add-on, not a re-ingestion), `sl_compile`'s pdflatex driver (the revise loop's *strategy* changes in Round 3, but the driver doesn't), `sl_verify_figures` (figure-key validation is fine — keep), F4's NOTES/EDIT routing (decoupling is fine and orthogonal to quality), F4.2's title/preamble editing (also orthogonal). ## Style configurability — the `SlideStyleProfile` contract

The gold methodology is the *default* profile, not the only one. The contract:

```python
@dataclass(frozen=True)
class SlideStyleProfile:
    name: str                                    # 'default' (gold), 'casual_talk', 'defense_long', ...
    theme: str                                   # 'Berlin', 'metropolis', 'Madrid', ...
    colortheme: str                              # 'dolphin', 'beaver', ...
    fonttheme: str                               # 'professionalfonts', 'serif', ...
    accent_color: tuple[int, int, int]           # RGB; defaults gold's (0, 90, 160)
    accent2_color: tuple[int, int, int]          # RGB; for highlight annotations
    closer_style: Literal['takeaway_open_q', 'thank_you', 'questions_only']
    bottlenecks_overview: bool                   # True for multi-paper, False for single
    references_qr: bool                          # True default; False to skip QR codes
    proposed_direction_placeholder: bool         # True when ≥2 papers; emit empty TikZ for user to fill
    audience_register: Literal['conference', 'master_cross_domain', 'casual']
    duration_minutes: int                        # drives word budget
    wpm_overrides: dict[str, int] | None         # per-language pace override
    watermark_path: str | None                   # path to logo image; None disables
    footer_id: str | None                        # e.g. "ID: 3-3"; appears in footline
```

Surfaces:
- **Default** comes from `slide_style_profiles.yaml :: default` (the gold spec encoded).
- **Per-session override** via deck-command chat turn → `_recompile_and_emit` with new profile.
- **Per-user persistent override** via active memory; `build_active_memory_block` injects "Use Traditional Chinese, metropolis theme, casual register" into the narrate prompt, which propagates into the emitted profile for that session.
- **Per-request override** via an optional `style_profile` field on `POST /chat` (later — not Round 1).

Rationale: this isolates "the gold methodology" as one *named profile* among many, instead of hardcoding it. The first audience that asks for *"don't end on take-away, just say thanks"* shouldn't trigger another plan round — they should flip `closer_style`.

## Done definition

F4.4 is done when ALL of:

1. **Developer-scoring gate (the harness):** Scorecard A scores ≥ 8/10 PASS on EACH of the 3 dev-set papers; Scorecard B scores ≥ 10/12 PASS on the multi-paper dev-set deck. These are *necessary* signals that the design rounds landed; they are *not* sufficient.
2. **User production gate (the real test, the user runs it):** The user picks their OWN set of varied test cases — single-paper, 2-paper, 3-paper, ≥5-paper; different topics (not just VLA/robotics); different languages — and runs each through the new pipeline on `:8000`. For each, the user judges *"is this a deck I could actually present?"*. F4.4 is done when the user says yes across the user's chosen test set. The harness's 3 papers do NOT cover this — generalization to unseen papers is exactly what the harness can't measure.
3. **Style override surface works end-to-end** — the user can flip closer style, theme, audience register, language via chat command or persistent memory, and the deck re-renders accordingly. Demonstrated on at least one user-chosen example.
4. **Real-API `:8000` confirmation** of 1–3 in the actual live stack (modelserver + MCP + backend + frontend), per CLAUDE.md's plan-phase completion gate.

This is *quality*-gated and *generalization*-gated. The harness ensures the design rounds were productive; the user's varied test set ensures the design generalizes. Round count is not predetermined. If a design passes the harness but fails on the user's first novel test paper, that's a Round N+1 signal — the design encoded the dev-set's idiosyncrasies, not the general principles.

**Agent-self-judge gate (precedes the user gate).** The user verification step (#2 above) is the FINAL gate. It is NOT invoked after every round. After each round, the agent compares its output to the gold samples side-by-side and decides honestly: *is this at the gold standard yet?* If no, design Round N+1 immediately — do not ping the user. Only when the agent's own honest comparison says *"yes, mine is now at-parity with the gold on both single-paper and multi-paper, notes pass the read-aloud test"* does the agent escalate to the user for #2–#4 verification. Premature escalation wastes the user's time on rounds that are obviously not done; the agent can judge sample-vs-creation directly without asking.

## Risk + mitigations

| Risk | Mitigation |
|---|---|
| Round 1 prompt rewrite makes the *default* (today's) behaviour regress on edge cases (e.g. single-paper decks) | The harness is the gate. We run BOTH the 3-paper case AND a 1-paper case (one of the three) every round. If the 1-paper output regresses, the prompt branched on N is wrong. |
| LLM ignores the new `pattern_kind` enum in `TalkOutline` and emits free-text patterns | Validate in `sl_narrate`'s schema; reject + retry once on schema-violation. Same pattern as today's TalkOutline JSON validation. |
| Berlin/dolphin theme breaks on systems without the required Beamer fonts | The compile loop already catches this. Default-profile theme is a YAML override away if it does. Document `professionalfonts` requirement (TeX Live `psnfss` package — already present in MikTeX full install). |
| `sl_audit_claims` halts the deck on a false positive | The audit is advisory: it writes findings to a `deck.audit_findings_json` field (new) but does NOT block emission. The frontend's deck card surfaces the count + a "review" button. Hard-block would mask the deck for legitimate compressions of notation. |
| The gold methodology assumes 5–10 minute talks. Lightning talks (3 min) and defenses (45 min) need different shapes | `SlideStyleProfile.duration_minutes` + `closer_style` cover the common cases. Edge cases (≥30 min defenses) get their own profile; not Round-1. |
| User feedback "the new pipeline is terrible" might mean *one specific thing* (e.g. just the closer, just the notes), not the whole pipeline | Round 1 IS gathering the diagnosis. The scorecard's per-dimension scoring resolves "what specifically is terrible". |

## Working-style notes (apply every round)

- **Agent self-judges first; escalate to human ONLY when agent is genuinely done.** This is the iron rule for the iteration loop. After each round, the agent (the worker — you) opens PaperHub's generated PDFs and the gold-sample PDFs side-by-side and judges *"is mine at the gold standard yet?"* — by your own reading, not by asking. As long as the answer is honestly "no, mine is still obviously weaker on dimension X", you do NOT ask the human to verify — you design Round N+1 and run it. Only when YOU are completely confident the output has reached the gold standard (single-paper parity with the per-paper gold AND multi-paper parity with the top-level gold AND clean speaker notes that pass the read-aloud test) do you ask the human for the final sign-off. Reason: a human gate after every round wastes the user's time on rounds where the gap is obvious; the user's job is the final verdict + the production test on novel papers (per the Done definition), not refereeing every iteration. If you find yourself reaching for "let me ask the user whether this is good enough" — and you can already see by direct comparison that it isn't — the answer is "no, do another round". Sample-vs-creation is something you can clearly judge yourself.
- **One lever per round.** If a round needs to change two things to be coherent (e.g. `pattern_kind` schema field requires both narrate AND draft to change in the same commit), they're the same lever and ship together — but only one *user-visible-quality* dimension is targeted.
- **Read the trace before writing any prompt fix** (agent-flow observability iron rule). Every `sl_*` step records its full input state in `tool_calls.args_redacted_json` and output in `result_summary_json`. *"The LLM is doing X"* without a trace quote is a hypothesis, not a diagnosis.
- **Enrich-then-diagnose** if the trace can't answer "why did `sl_narrate` skip the bottlenecks slide" — first commit is enriching `record_result` (e.g. log which pattern types it considered + rejected); second commit fixes.
- **Commit messages**: Conventional Commits per `CLAUDE.md`. Scope `slides` for prompt/topology; `style` for profile surface; `harness` for the calibration scripts. Example: `feat(slides): rewrite narrate to plan named patterns (F4.4 Round 1)`.
