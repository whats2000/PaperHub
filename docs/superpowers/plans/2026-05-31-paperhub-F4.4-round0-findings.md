## The OLD pipeline's actual runtime (corrected for what I'd missed from prompts-alone)

The plan's first cut of the contract gap was based on reading prompts only. Reading `core.py` revealed runtime details that change the picture:

1. **The OLD pipeline is THREE-STAGE, not single-call.** `_generate_slides_with_stages` runs:
   - Stage 1: `initial` prompt ‚Üí full `slides.tex` (one LLM call, sees whole paper + figure paths)
   - Stage 2: `update` prompt ‚Üí re-read the just-emitted `slides.tex` + paper + figure paths, enforce ‚â§15-words-per-item + split-overflow + figure-path correction + ampersand-escape + strip `\cite` (another LLM call)
   - Stage 3: `try_compile_with_fixes` loop ‚Äî pdflatex compile, if it fails feed the ChkTeX output into the `revise` prompt, up to 3 retries
   
   So the OLD project's actual quality contract has **two LLM passes before compilation** + a ChkTeX-driven revise loop. Stage 2 is a dedicated quality-discipline pass ‚Äî distinct from syntax-revise. PaperHub today has no equivalent.

2. **`ADDITIONAL.tex` flow is load-bearing.** Three steps:
   - `extract_definitions_and_usepackage_lines(latex_source)` walks the paper's preamble.
   - `build_additional_tex(defs_pkgs)` writes the paper's `\newcommand` / `\renewcommand` / `\usepackage` lines into a sidecar `ADDITIONAL.tex`.
   - `add_additional_tex(latex_source)` ensures `\input{ADDITIONAL.tex}` is in the slides preamble.
   
   The `initial` prompt's preamble template includes `\input{ADDITIONAL.tex}` as a fixed line. Slides inherit every macro the paper defined ‚Äî so math like `\E_{x \sim p}[\cdot]` or `\R^d` renders correctly. **PaperHub's F2 PaperAsset does NOT preserve these.** Any paper-specific macro in a slide's equation will silently break compilation, OR ‚Äî worse ‚Äî produce a different-than-the-paper rendering (e.g. `\R` may resolve to a different default if the paper redefined it).

3. **`copy_image_assets_from_cache` + `find_image_files`** ‚Äî figures from the source are copied into the workspace; then a directory walk produces a CONCRETE LIST of every allowed image path. The `initial` prompt embeds `{figure_paths}` verbatim ("Here is the list of image paths you are allowed to use: ..."). The LLM picks from a fixed allowlist. PaperHub F3 does this via the `FIGURE INVENTORY` block + `sl_verify_figures` (which deterministically rewrites unknown keys to `[figure omitted]`) ‚Äî that part is roughly equivalent, modulo a different key scheme (`p{idx}-{figure_id}` vs raw filename).

4. **`generate_speaker_notes` is a SINGLE LLM call for the whole deck.** It receives the annotated beamer (overlay frames marked) + the full original paper LaTeX, and emits `[SLIDE N]` blocks for ALL slides at once. A regex splits them. Then `edit_speaker_notes` is a second-pass refinement that takes the prior `speaker_notes.json` + the current beamer (which may have changed via edits) and re-matches notes to slides BY CONTENT (not by index). PaperHub today calls `slides_note_author_v1` per-slide ‚Äî each call sees one frame's TeX, has no cross-deck context, so cross-slide connectors ("Remember from slide 3‚Ä¶") cannot exist.

5. **The `update` prompt has the split-on-overflow contract baked in.** Direct quote: *"If a slide has too much text overall or items are still too long after rewriting, split it into multiple frames."* Combined with `interactive_edit_single_slide`'s *"If the instruction asks to split the frame, you MUST return ALL the new frames"* ‚Äî splitting is a first-class strategy. PaperHub's `sl_revise` reacts to Overfull warnings by asking the LLM to shrink content; it does not split.

6. **`_shift_speaker_notes_after_frame_edit` does the bookkeeping when a frame edit changes the frame count.** A frame containing PDF pages `[lo, hi]` is replaced; notes for those pages are dropped (the env content is gone); notes after `hi` shift by `delta = new_count - old_count`. This is sophisticated and PaperHub's F4 `edit_frame` flow doesn't have it (F4 simply preserves notes by `slide_index` and drops surplus if the count changes ‚Äî the plan acknowledges this as a known limitation).

## Round-1 hypothesis ‚Äî REVISED: agentic-brief + bounded-fallback hybrid

Initial pick was candidate (i) "per-paper sub-deck then merge". On user review the **context-size problem** with (i) surfaced: one ML paper's LaTeX is 30‚Äì80k tokens; 3 concatenated + figure inventory + prompt + output budget exceeds 200k. paper2slides-plus gets away with monolithic-context because it's single-paper AND tolerates occasional failure on giant papers; PaperHub at multi-paper scale cannot. Updated hypothesis is **a refined candidate (iv) hybrid** that gives every downstream stage bounded, focused context:

```
sl_resolve
  ‚Üí sl_paper_brief    (FAN-OUT per paper: reuses paper_qa_subagent ‚Äî navigates
                       the paper via list_sections / read_section /
                       read_figure_block / read_equations tools ‚Äî emits
                       PaperTalkBrief: contribution, method core, top 3-5
                       figures with interpretations, top 3-5 quantified
                       results, key equations verbatim, custom \newcommand
                       lines, talk-shape hint)
  ‚Üí sl_plan_deck      (ONE call: takes N PaperTalkBriefs + style profile +
                       budget ‚Üí outlines the multi-paper deck with pattern_kind
                       per slide, slide-to-paper attribution, per-slide goal /
                       figure / equation choice. For N=1 this degenerates to a
                       single-paper outline. Total brief context ~25k for 3
                       papers ‚Äî fits easily.)
  ‚Üí sl_render_slide   (PER-SLIDE FAN-OUT: each slide knows its pattern_kind,
                       its assigned figure/equation/result, its paper_id;
                       renders the TeX. CAN call back into the per-paper
                       subagent via read_section / read_figure_block when the
                       brief is insufficient ‚Äî capped at 1-2 callbacks per
                       slide to bound cost. This is Option A as a fallback
                       inside Option B.)
  ‚Üí sl_coherence ‚Üí sl_assemble ‚Üí sl_verify_figures ‚Üí sl_compile ‚Üí
    sl_notes_finalize ‚Üí sl_emit   (mostly unchanged)
```

User-confirmed decisions (2026-05-31, this round):
- **Brief source**: reuse `paper_qa_subagent` (proven Plan-C-v2.10 pattern, full agent-flow trace recorded automatically).
- **Render fallback**: bounded callback enabled (1-2 `read_section`/`read_figure_block` calls per slide max) so the brief doesn't have to be 100% self-sufficient.
- **Baseline timing**: harness baseline runs BEFORE Round-1 design lands, so Round-1's diff has a concrete starting point.

**Why this over the plan's other three candidates:**

- **vs (i) "per-paper sub-deck then merge"**: (i) needs the full paper LaTeX in the per-paper call. For long papers this blows context or fails silently. The brief approach extracts what the talk needs from the paper agentically, keeping every call bounded.
- **vs (ii) "plan-then-render"**: similar to ours but plan/render see only briefs, no agentic read. The brief stage IS the missing intelligence layer ‚Äî it's what makes a single brief actually carry enough signal to plan against.
- **vs (iii) "fragmented + restored context"**: keeps today's frame-by-frame fragmentation but passes all briefs everywhere. Cheaper per call than (iv) but loses the agentic read; long papers' contextual structure (e.g. which equation is referenced where) is invisible to the planner.

**Trade-offs:**

- 3 LLM stages per deck instead of 2 (brief + plan + per-slide-render), but each is FOCUSED ‚Äî no node carries the full paper text.
- `sl_paper_brief` is an agentic call (multiple tool round-trips per paper) ‚Äî heavier than the current `sl_understand`. Net token cost is similar to today's `sl_understand` if the brief is well-targeted; can be higher if the subagent over-reads.
- The render stage is closer to today's `sl_draft` than to paper2slides-plus's monolithic call. We lose paper-level cross-slide coherence WITHIN a paper, but `sl_coherence` already does a whole-deck pass to recover that.
- We do NOT get OLD-style "the whole paper informs every slide" ‚Äî but that was never going to work at multi-paper scale anyway.
- Briefs are CACHEABLE per `paper_content_id` (one paper, one brief ‚Äî re-use across sessions/decks). The OLD project's `\input{ADDITIONAL.tex}` plumbing maps to one field on the brief, computed once.

**Why this over the other three:**
- Candidate (i) is the ONLY one that restores OLD-style monolithic-context contract clarity *per paper*. The OLD project's `stages.initial`+`stages.update` two-call pattern translates directly into one `sl_per_paper_deck` node that runs both calls back-to-back per paper. Every density rule the OLD prompt was specific about (‚â§15-words, ‚â§4-items, split-on-overflow, figure-only-frame, equation+itemize, no-Thank-you, ADDITIONAL.tex, abbreviations) goes into ONE prompt that has ONE paper's full context.
- For multi-paper output, `sl_merge_multi_paper` does the connective-tissue work ‚Äî bottlenecks-overview table (or 3-row visual), proposed-direction placeholder, take-away+open-question closer. It sees ALL N mini-decks + a cross-paper brief and emits the final deck. This work has no OLD-project equivalent (OLD is single-paper); it's new for F4.4 and naturally lives in its own LLM call.
- Single-paper case: the merge stage is a degenerate no-op (or a light pass-through) ‚Äî N=1 means the per-paper mini-deck IS the final deck. So single-paper PaperHub falls through to roughly the OLD project's pipeline, preserving its quality.
- Candidate (ii) "plan-then-render" keeps a planning stage but loses the per-paper monolithic-context win on the RENDER call (the LLM still has to render N papers' worth of slides from BRIEFS, not full LaTeX).
- Candidate (iii) "fragmented topology + restored context" passes the test of "every node sees enough state" but keeps the per-frame fragmentation ‚Äî each draft call still emits one slide at a time, losing the unifying coherence the OLD project's monolithic call had.
- Candidate (iv) "hybrid" is over-engineered for Round 1; if (i) under-performs on the merge stage we can split it later.

**Cost of (i):** 2 LLM calls per paper (initial + update) + 1 merge call = `2N + 1` large LLM calls per deck, vs the current pipeline's `2N + 1` calls of roughly the same total volume (N understand + 1 narrate + N√óslides draft + 1 coherence). Token count is similar; the difference is concentration (one big call per paper instead of many small ones). Quality per token should be much higher.

**Risks specific to (i):**
- Larger per-call payload may hit token limits for long papers ‚Äî need fallback (chunk the paper or fall back to PaperAsset summary).
- Two-stage per-paper (initial + update) doubles per-paper latency ‚Äî acceptable if quality lifts.
- Merge stage is novel ‚Äî quality of multi-paper synthesis depends on the merge prompt's contract clarity. This is exactly where the academic-talk-deck.skill's named patterns become the merge prompt's instruction set.

## Round-1 lever (the ONE thing to change in Round 1)

Implement the agentic-brief + bounded-fallback hybrid topology end-to-end. Concretely:

1. **New schema `PaperTalkBrief`** (richer than today's `PaperBrief`): `contribution` + `method` + `key_results` (with benchmark names + numbers) + `key_figures` (each: key, role-in-talk, 1-line interpretation) + `key_equations` (verbatim LaTeX + 1-line role) + `paper_newcommands` (raw text block for `ADDITIONAL.tex` plumbing) + `talk_shape_hint` (e.g. "2 slides: concept + math" vs "3 slides: concept + math + results table").
2. **New prompt `slides_paper_brief_v1.yaml`** + **new node `sl_paper_brief`** ‚Äî fans out per paper, dispatches to a `paper_qa_subagent`-style agentic reader with tools `list_sections` / `read_section` / `read_figure_block` / `read_equations` (already exist in `paper_qa` infra). Each subagent emits one `PaperTalkBrief`. Briefs are CACHED per `paper_content_id` so re-runs and cross-session reuse are free.
3. **New prompt `slides_plan_deck_v1.yaml`** + **new node `sl_plan_deck`** ‚Äî consumes N `PaperTalkBrief`s + `SlideStyleProfile` + budget ‚Üí emits the multi-paper outline. Each planned slide has a `pattern_kind` (`title`, `references_qr`, `motivation_figure`, `bottlenecks_table`, `concept_2col`, `math_stack`, `proposed_direction_placeholder`, `plan_numbered`, `takeaway_closer`, etc.) + `paper_id` attribution + assigned `figure_key`/`equation`. For N=1 the prompt branches to a single-paper outline (no bottlenecks-table, no proposed-direction).
4. **New prompt `slides_render_slide_v1.yaml`** (or one prompt per `pattern_kind`) + **new node `sl_render_slide`** ‚Äî per-slide fan-out. Each call receives ONE slide's plan + the assigned brief subset. CAN call back into the per-paper subagent via `read_section`/`read_figure_block` (1-2 callbacks max per slide) when the brief field is insufficient. Replaces today's `sl_draft`.
5. **`sl_assemble` change** ‚Äî concat each paper's `paper_newcommands` (from the briefs) into the slides preamble (de-dup + collision-rename when N>1). Closes the OLD project's `\input{ADDITIONAL.tex}` gap without needing a separate sidecar file.
6. **Replace in subgraph**: `sl_understand` + `sl_narrate` + `sl_draft` ‚Üí `sl_paper_brief` + `sl_plan_deck` + `sl_render_slide`.
7. **Keep unchanged for Round 1**: `sl_coherence`, `sl_verify_figures`, `sl_compile` (with its revise loop), `sl_notes_finalize`, `sl_emit`. These will be revisited if Round-1 scorecards point at them.

This is ONE topology lever (per the one-lever-per-round rule). The seven sub-pieces above are coherent ‚Äî they ship together. Subsequent rounds attack what Round-1 scorecards reveal as the next-lowest-scoring dimensions (forecast: per-pattern render prompts in detail, then whole-deck-notes call + speakable post-processor, then ChkTeX-driven revise loop, then claim-audit, then `SlideStyleProfile` surface).

**Subagent dispatch shape (per the subagent-driven-development flow):** Round 1 is decomposed into per-task subagents, each with its own commit + spec-reviewer + code-quality-reviewer pass before the next task starts:
- T1: `PaperTalkBrief` schema + `slides_paper_brief_v1.yaml` + `sl_paper_brief` node + tests.
- T2: `slides_plan_deck_v1.yaml` + `sl_plan_deck` node + outline schema with `pattern_kind` + tests.
- T3: `slides_render_slide_v1.yaml` + `sl_render_slide` node + per-`pattern_kind` template branching + bounded-callback wiring + tests.
- T4: `sl_assemble` change to plumb `paper_newcommands` into the preamble + tests.
- T5: Subgraph wiring (replace old nodes) + integration test.
- T6: Harness re-run + scorecards + self-judge.
