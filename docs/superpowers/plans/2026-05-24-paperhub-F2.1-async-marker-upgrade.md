# Plan F2.1 â€” Marker as an Optional Asynchronous Add-On

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make PDF ingestion return in seconds on a PyMuPDF baseline that is fully functional without Marker, and upgrade to Marker-quality assets + chunks via a durable, eager, strictly-sequential background worker â€” so Marker is a preferred-when-present add-on, never a hard dependency or a UX blocker.

**Architecture:** Synchronous live ingest (PyMuPDF: text + heading sections â†’ chunk + embed + degraded `PaperAsset` + HTML) sets `paper_content.asset_status` to `'pymupdf_only'` or, when the `marker` service is reachable, `'marker_pending'`. A single in-process background worker (one paper at a time â†’ no GPU OOM; resumes on startup) drains `marker_pending` papers: Marker `/extract` â†’ upgrade the on-disk `PaperAsset` + re-chunk + re-embed off Marker's structure â†’ `'marker_ready'` (failure â†’ `'marker_failed'`, baseline retained). arXiv e-print papers stay synchronous + high-fidelity (`'latex'`), never enqueued. Consumers read whatever asset is on disk and benefit automatically when a paper flips to `marker_ready`.

**Tech Stack:** Python 3.11 / `uv`, FastAPI lifespan, aiosqlite, existing `PaperPipeline`, `MarkerClient`, `marker_to_asset`, `chunker`, `embedder`, `ChromaStore`. SRS **v2.20** Â§III-5.1 / Â§III-6 / Â§III-7.

**Depends on:** F2 (shipped â€” `PaperAsset`, `MarkerClient`, `marker_to_asset`, `latex_to_asset`, `paperhub-backfill-assets`). **Must land before F3** consumes the asset (F3 reads whatever quality is on disk; this plan guarantees it is at least the PyMuPDF baseline and upgrades silently).

## Addendum A1 â€” Organized-markdown chunks + highlight-safe `match_text`

**Motivation:** the v2.20 worker re-chunks PDFs from `_marker_text_and_sections`, which *skips figure blocks* and concatenates block plain-text â€” so figure captions never reach RAG and the structure is flat. Chunking over Marker's **organized markdown** (headings, figure captions, tables, equations in reading order) gives the answering model + embedder far richer context. BUT the Citation Canvas locates a chunk in the source PDF (and in the HTML text-search fallback) by a **start-anchored normalized prefix** of `chunk.text` (`buildTargets` in `frontend/src/lib/findAndHighlight.ts`, used by `locatePassage` in `pdfHighlight.ts`). Markdown markers at a chunk's start (`## `, `[Figure â€¦]`, `| `, `$$`) poison every prefix â†’ the PDF text layer never matches â†’ the highlight breaks. The PDF view has NO sentinel fallback (it's text-match only), and after a marker upgrade chunks carry `dom_id=None`, so the HTML view also falls back to text-search â€” both depend on matchable text.

**Decision (A1): decouple model-context text from the highlight anchor.** Each chunk stores TWO views:
- `text` â€” organized markdown (embedded + read by the answering model).
- `match_text` â€” clean reading-order plain text (markdown stripped) that the resolver matches against the PDF/HTML. NULL for non-Marker chunks (resolver falls back to `text` â€” unchanged behavior).
