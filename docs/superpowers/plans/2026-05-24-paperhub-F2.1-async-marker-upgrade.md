# Plan F2.1 — Marker as an Optional Asynchronous Add-On

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make PDF ingestion return in seconds on a PyMuPDF baseline that is fully functional without Marker, and upgrade to Marker-quality assets + chunks via a durable, eager, strictly-sequential background worker — so Marker is a preferred-when-present add-on, never a hard dependency or a UX blocker.

**Architecture:** Synchronous live ingest (PyMuPDF: text + heading sections → chunk + embed + degraded `PaperAsset` + HTML) sets `paper_content.asset_status` to `'pymupdf_only'` or, when the `marker` service is reachable, `'marker_pending'`. A single in-process background worker (one paper at a time → no GPU OOM; resumes on startup) drains `marker_pending` papers: Marker `/extract` → upgrade the on-disk `PaperAsset` + re-chunk + re-embed off Marker's structure → `'marker_ready'` (failure → `'marker_failed'`, baseline retained). arXiv e-print papers stay synchronous + high-fidelity (`'latex'`), never enqueued. Consumers read whatever asset is on disk and benefit automatically when a paper flips to `marker_ready`.

**Tech Stack:** Python 3.11 / `uv`, FastAPI lifespan, aiosqlite, existing `PaperPipeline`, `MarkerClient`, `marker_to_asset`, `chunker`, `embedder`, `ChromaStore`. SRS **v2.20** §III-5.1 / §III-6 / §III-7.

**Depends on:** F2 (shipped — `PaperAsset`, `MarkerClient`, `marker_to_asset`, `latex_to_asset`, `paperhub-backfill-assets`). **Must land before F3** consumes the asset (F3 reads whatever quality is on disk; this plan guarantees it is at least the PyMuPDF baseline and upgrades silently).

## Addendum A1 — Organized-markdown chunks + highlight-safe `match_text`

**Motivation:** the v2.20 worker re-chunks PDFs from `_marker_text_and_sections`, which *skips figure blocks* and concatenates block plain-text — so figure captions never reach RAG and the structure is flat. Chunking over Marker's **organized markdown** (headings, figure captions, tables, equations in reading order) gives the answering model + embedder far richer context. BUT the Citation Canvas locates a chunk in the source PDF (and in the HTML text-search fallback) by a **start-anchored normalized prefix** of `chunk.text` (`buildTargets` in `frontend/src/lib/findAndHighlight.ts`, used by `locatePassage` in `pdfHighlight.ts`). Markdown markers at a chunk's start (`## `, `[Figure …]`, `| `, `$$`) poison every prefix → the PDF text layer never matches → the highlight breaks. The PDF view has NO sentinel fallback (it's text-match only), and after a marker upgrade chunks carry `dom_id=None`, so the HTML view also falls back to text-search — both depend on matchable text.

**Decision (A1): decouple model-context text from the highlight anchor.** Each chunk stores TWO views:
- `text` — organized markdown (embedded + read by the answering model).
- `match_text` — clean reading-order plain text (markdown stripped) that the resolver matches against the PDF/HTML. NULL for non-Marker chunks (resolver falls back to `text` — unchanged behavior).

## Addendum A2 — Marker-native layout-preserving markdown chunks + per-chunk page (SUPERSEDES A1's assembler)

**Why (root cause from a live test):** asking "what in table 1" of the Transformer paper failed. The trace (run 180) showed the agent navigated correctly — the failure was PARSING: A1's `marker_doc_to_markdown` rendered tables with `strip_html(block.html)`, **flattening** `| Layer | Complexity | … |` into a run-on string (`Layer TypeComplexity per Layer…O(n²·d)`), **duplicating** it (the service `_flatten` emits the Table parent AND its child cells), and **mis-filing** it into the wrong section. Agentic RAG has the model READ the chunk, so destroyed layout = unanswerable — for tables, figures, and any layout element. **Fix: use Marker's NATIVE markdown renderer** (it emits real `| … |` markdown tables in correct reading order) as the chunk source; keep the JSON blocks for the `PaperAsset` (slides).

**Keeps from A1:** `chunks.match_text` column, `strip_markdown` (now applied to native markdown), the frontend `resolveNeedle` (`match_text ?? text`). **Replaces:** `marker_doc_to_markdown` (delete) — the service now provides the markdown.
