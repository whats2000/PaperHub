# Plan F2.1 ‚Äî Marker as an Optional Asynchronous Add-On

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make PDF ingestion return in seconds on a PyMuPDF baseline that is fully functional without Marker, and upgrade to Marker-quality assets + chunks via a durable, eager, strictly-sequential background worker ‚Äî so Marker is a preferred-when-present add-on, never a hard dependency or a UX blocker.

**Architecture:** Synchronous live ingest (PyMuPDF: text + heading sections ‚Üí chunk + embed + degraded `PaperAsset` + HTML) sets `paper_content.asset_status` to `'pymupdf_only'` or, when the `marker` service is reachable, `'marker_pending'`. A single in-process background worker (one paper at a time ‚Üí no GPU OOM; resumes on startup) drains `marker_pending` papers: Marker `/extract` ‚Üí upgrade the on-disk `PaperAsset` + re-chunk + re-embed off Marker's structure ‚Üí `'marker_ready'` (failure ‚Üí `'marker_failed'`, baseline retained). arXiv e-print papers stay synchronous + high-fidelity (`'latex'`), never enqueued. Consumers read whatever asset is on disk and benefit automatically when a paper flips to `marker_ready`.

**Tech Stack:** Python 3.11 / `uv`, FastAPI lifespan, aiosqlite, existing `PaperPipeline`, `MarkerClient`, `marker_to_asset`, `chunker`, `embedder`, `ChromaStore`. SRS **v2.20** ¬ßIII-5.1 / ¬ßIII-6 / ¬ßIII-7.

**Depends on:** F2 (shipped ‚Äî `PaperAsset`, `MarkerClient`, `marker_to_asset`, `latex_to_asset`, `paperhub-backfill-assets`). **Must land before F3** consumes the asset (F3 reads whatever quality is on disk; this plan guarantees it is at least the PyMuPDF baseline and upgrades silently).

### Task 5: Durable sequential background Marker worker

**Files:** Create `marker_worker.py`; modify `app.py` lifespan; Test `tests/test_marker_worker.py`.

Core (testable, no live service): `async def upgrade_one(pcid, *, conn, chroma, marker_client, max_pages) -> str` ‚Äî load source for `pcid`; Marker `extract` (max_pages); `marker_doc_to_asset` ‚Üí `write_paper_asset` (overwrite); re-derive text+sections via `_marker_text_and_sections`; re-chunk + re-embed + replace `chunks` rows + Chroma vectors (reuse the exact delete-then-insert sequence from `cli/reingest._reingest_one`, including `sections_json` recompute); set `asset_status='marker_ready'`; on any exception set `'marker_failed'` and return that. **Strictly one paper at a time.**

Loop: `async def run_worker(stop: asyncio.Event)` ‚Äî query `SELECT id FROM paper_content WHERE asset_status='marker_pending' ORDER BY id`; for each, `await asyncio.to_thread(... upgrade_one sync core ...)` (Marker call + IO off the loop); sleep-poll (e.g. `await asyncio.wait_for(stop.wait(), timeout=N)`) when the queue is empty; exit cleanly on `stop`. **Startup resume is automatic** ‚Äî `marker_pending` rows from a previous process are picked up on first scan.

### Task 7: Marker low-VRAM / text-layer tuning (so upgrades succeed on 6 GB)

**Files:** Modify `marker_service/app.py` + `docker-compose.yml` (env); no Python-unit test (config + a real-run check in Task 8). **Motivation:** the F2.1 live test showed a dense two-column page runs full Surya OCR (200+ "Recognizing Text" lines) that fills ~5.9 GB and **crashes the worker process** even at `max_pages=1`. Most PaperHub PDFs (arXiv-via-pdf, publisher uploads) have **clean embedded text layers**, so most of that OCR is avoidable.

- [ ] **Step 1 ‚Äî reduce peak VRAM via Surya batch-size env.** Marker/Surya read batch sizes from env: set conservative low-VRAM defaults in `docker-compose.yml` (overridable), e.g. `RECOGNITION_BATCH_SIZE=8`, `DETECTOR_BATCH_SIZE=4`, `LAYOUT_BATCH_SIZE=4`, `TABLE_REC_BATCH_SIZE=4`, `OCR_ERROR_BATCH_SIZE=4` (confirm exact names against the installed marker/surya version ‚Äî they are documented in surya's settings). Goal: peak VRAM stays well under 6 GB on a dense page.
- [ ] **Step 2 ‚Äî trust the text layer when present.** In `_converter`, prefer the PDF's embedded text over re-OCR for digital-born PDFs: do NOT set `force_ocr`; investigate the installed Marker's OCR-avoidance knobs (e.g. `disable_ocr` / `strip_existing_ocr=False` / raising the OCR-error threshold so a good text layer is not flagged for re-OCR). Keep equation/figure/table block detection (that's the value-add); only the wholesale line-level OCR is what we want to skip when `pdftext` already has the text. Document the chosen knobs in an app.py comment.
- [ ] **Step 3 ‚Äî rebuild + measure.** `docker compose build marker && docker compose up -d marker`; re-run the dense paper (pcid=22) via `paperhub-backfill-assets --paper-content-id 22 --force` and confirm it (a) completes without the worker crashing, (b) in a few minutes not tens, (c) still yields real figures + captions + equations. If a clean text-layer-trust knob isn't available in this Marker version, the batch-size reduction alone must at least stop the crash (slow-but-survives is acceptable; crash is not).
- [ ] **Step 4 ‚Äî commit** `perf(slides): low-VRAM Surya batch sizes + text-layer trust for Marker (F2.1)`.
