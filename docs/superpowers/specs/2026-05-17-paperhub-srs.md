# PaperHub

**A UX-first chat shell with its own paper knowledge base + RAG + slide pipeline, assembled by adapting utility functions from two reference projects.**

| Field | Value |
| --- | --- |
| Version | v2.17 |
| Date | May 2026 |
| Posture | **UX-first + decomposed reuse.** PaperHub owns its data layer, RAG pipeline, and slide pipeline so every step is traceable. Utility functions are *copied and adapted* from two reference projects rather than consumed as services. |
| Reference projects (utility source) | [`paper2slides-plus`](https://github.com/whats2000/paper2slides-plus) — extraction utils for arXiv / PDF / LaTeX projects + LaTeX helpers, **decomposed; not run as a service** · [`Intro2GenAI-hw1`](https://github.com/whats2000/Intro2GenAI-hw1) — chat-shell + MCP UX idioms ported to React |

# Part 1 — 需求規格書 (Software Requirements Specification)

## I-1. 專案背景 (Project Background)

Two reference projects exist in the author's portfolio that PaperHub draws from:

- **`paper2slides-plus`** — a FastAPI service that turns arXiv IDs / PDFs / LaTeX-zips into Beamer slide decks, with a chat-with-paper Q&A mode. **Decomposed, not run as a service.** Its extraction utilities (arXiv source download, PDF text extraction, LaTeX project handling) are copied into PaperHub and adapted; its chat-with-paper module is *not* reused because it cannot surface chunk-level citation IDs for the Citation Canvas; its slide-pipeline orchestrator is *not* reused because a modern-framework rewrite is needed (§III-6).
- **`Intro2GenAI-hw1`** — a chat UI (Node + Express + vanilla JS) that streams via SSE, manages history, forks conversations, renders thinking blocks, and integrates MCP tools. Its UX idioms are ported to React; its Node backend is not deployed.

PaperHub is therefore a self-contained system: a chat surface, a router, four agents, an in-repo paper knowledge base with RAG, an in-repo slide pipeline, and a tool-call audit log. The four product dimensions PaperHub commits to are problem classification, tool selection, model answer-quality comparison, and traceability — see §I-5 (functional requirements) and §I-8 (acceptance criteria) for the concrete shape of each.

**Engineering principle.** Invest in UX; reuse code by *copying and adapting*, not by service integration. The chat shell, citation canvas, routing badge, trace panel, and LangGraph router are new code; the paper-extraction utilities are copied + edited from `paper2slides-plus`; the chat-shell layout idioms are copied + ported from `Intro2GenAI-hw1`. Every step of the pipeline is in-repo so every trace row points at code we control.

## I-2. 問題描述 (Problem Description)

Today's chat interfaces to LLMs hide three things this product makes first-class:

1. **Opaque tool / model selection.** A general-purpose chat can answer a paper question, run an SQL aggregation, or call an external tool — but the user cannot tell whether the agent picked the right capability or whether a wrong answer was caused by the model, the prompt, or a wrong tool firing.
2. **Ungrounded citations.** When an LLM says *"according to §3.2"*, there is no path from the cited token back to the actual paragraph in the source paper. For research use this is fatal.
3. **Non-reproducible agent runs.** Without a structured per-step audit trail (model, tool, arguments, latency, status), multi-agent answers cannot be debugged or compared across model choices.

PaperHub addresses all three **at the UI surface**: a **routing badge** makes (1) visible per turn; a **citation canvas** with click-to-scroll-and-highlight makes (2) inspectable; a **trace panel** sourced from a `tool_calls` SQLite table makes (3) replayable. The backend doing the actual research work is reused from existing services — the contribution is the inspection layer on top.

## I-4. 使用情境 (Use Cases — 6)

The reading flow is intentionally split into three explicit moments: **search**, **curate**, **discuss / generate**. The user always sees a stable shortlist of "reference sources" attached to the current chat session, and can toggle individual ones on / off — every downstream Q&A or slide generation honours that toggle.

### UC-1 — Search arXiv and shortlist candidates
*"Find recent papers on Mixture-of-Experts routing."* → Router picks `paper_search` → Research Agent runs a **read-only tool-calling loop** (v2.4 — read-only palette, shortlist + optional finalize markers): per turn it decides whether to (a) ask a clarifying question when the user intent is too vague to search (e.g. *"find me good ML papers"* → *"What problem are you trying to solve — routing, training stability, or something else?"*), (b) call `search_library(query)` against the user's deduplicated indexed library, (c) call `search_semantic_scholar(query, max_results)` against `/graph/v1/paper/search` (v2.4 — was `search_arxiv` in v2.3), or (d) call `search_semantic_scholar` again with a refined query (capped at N=3 calls per turn) before shortlisting. The agent's final assistant message names a **shortlist of 3-5 best candidates** with a one-line reason per pick + an optional `finalize: true` marker on the most-confident 1-2 picks. The chat endpoint **auto-attaches finalize-flagged candidates** (cap 2 enforced server-side; calls `add_paper_to_session_dispatch` → triggers download or cache-hit), then emits a `search_results` SSE event with the structured candidate list including `finalize`/`auto_added`/`papers_id`/`error` fields. The frontend renders each as a card with title / abstract / authors / year: auto-added rows show "Added by agent ✓" with toggle/remove controls; suggested-only rows show an **Add as reference** button. **Suggested-only papers are not downloaded** — only finalized (agent-auto) + user-clicked (Add card) + Library-Browser-attached papers trigger ingestion; cache-hit if already indexed. Multi-turn refinement across user replies is supported by the same agent reading prior `AgentState.history`; the user can answer the clarifying question in the next chat turn and the agent picks up the refined intent.

### UC-2 — Promote a paper to a session reference (download-if-needed or add-from-library)
The user clicks **Add as reference** on one or more search results → the Paper Pipeline (§III-5.1) runs. The pipeline's **first step is a cache lookup** on `content_key` (e.g. `arxiv:2403.01234`): if a matching `paper_content` row already exists (from a previous session), the pipeline **skips** download, extraction, HTML render, chunking, and embedding entirely, and just inserts a new `papers (session_id, paper_content_id, enabled=true, added_at)` row pointing at the existing content. On cache miss, the pipeline downloads, extracts, renders HTML, chunks, embeds, persists a `paper_content` row + `chunks` + Chroma vectors **once**, then inserts the `papers` row. Either way the paper appears in the **Reference Sources** sidebar panel under the current chat session, marked enabled by default. The user can toggle any reference on / off; toggling does *not* delete anything, it only flips `papers.enabled` for that row.

The user can also bypass the search step entirely by browsing their **already-indexed library** (the union of `paper_content` rows across all of their prior sessions) and attaching any of those papers to the current session — this is the cache-hit path surfaced as a first-class action (v2.3). The backend exposes `GET /papers/library` (paginated list of `paper_content` rows not yet attached to the current session) and `POST /papers/from-library` (inserts a fresh `papers (session_id, paper_content_id, enabled=true, added_at)` row referencing an existing `paper_content_id`; idempotent on `UNIQUE (session_id, paper_content_id)`). Outcome is identical to UC-2's cache-hit path — no download, no re-extraction, no re-embedding.

**Ingestion source selection** (v2.4 — runs through `add_paper_to_session_dispatch(paper_id)` regardless of the trigger): three entry points share the dispatcher — (1) the chat endpoint auto-attaches finalize-flagged candidates from the agent's shortlist (max 2 per turn), (2) `POST /papers` (user clicks Add on a SearchResultList card), (3) `POST /papers/from-library` (user attaches from the Library Browser drawer; always cache-hit by definition). For `paper_id="ss:<paperId>"` the dispatcher fetches the paper's metadata and chooses the download path: (a) if `externalIds.ArXiv` is present, recurse into the arXiv-source ingestion path (LaTeX source is the highest-fidelity ingest — better chunking, math intact, figures resolvable); (b) else if `openAccessPdf.url` is present, fetch the PDF via httpx and run the PyMuPDF extraction path with `kind='pdf_upload'` and `sha256(<pdf-bytes>)` as the `content_key`; (c) else raise `NoIngestibleSourceError` — chat endpoint marks the candidate as `auto_added=false, error="no_ingestible_source"` so the UI can grey out the Add button; `POST /papers` returns HTTP 422 with the same fields. The arXiv-source path is **always preferred over the PDF path** when both are available because LaTeX preserves structure that PDF text extraction loses. The canonical `content_key` (`arxiv:<id>` if arXiv-backed, `sha256:<hex>` for PDF-only) means the same paper picked via any entry point cache-hits future invocations. **Suggested-only candidates (no finalize flag, no user click) are not downloaded** — they remain metadata in the SSE event payload only.

### UC-3 — Multi-paper Q&A with the Citation Canvas *(headline)*
*"How do these two papers differ on expert collapse?"* → Router picks `paper_qa` → Research Agent runs the **v2.10 agentic hierarchical pipeline** (§III-5.2) over the **currently-enabled references**: one per-paper subagent per reference navigates that paper's section table-of-contents (`list_sections` → `read_section`), picks the chunks bearing on the question, and cites them; a flagship finalizer reads the raw cited chunks across papers and generates an answer with inline `[chunk:<id>]` markers tied to real `chunks.id` rows → clicking one opens the **Citation Canvas** on the right, scrolls to the chunk, highlights it. Disabling a reference and re-running the same question dispatches a subagent only for the remaining enabled set.

### UC-4 — Slide generation from the enabled reference set
*"Make slides comparing these references."* → Router picks `slides` → Report Agent reads the same enabled set → runs the in-repo Slide Pipeline (§III-5): structure planning over the chunk corpus → per-section LLM generation → emit modern-framework deck (Marp / Slidev / Beamer-via-LangGraph — choice deferred to implementation, see §III-5) → return artefact path as a downloadable chip in the chat. The pipeline reuses LaTeX helper functions copied from `paper2slides-plus` but is orchestrated by new code. Toggling a reference off before generation excludes it from the deck.

### UC-5 — Library-stats NL2SQL (read-and-act)
*"How many papers did I add this week? Which sessions reference paper X?"* → Router picks `library_stats` → SQL Agent introspects the schema via the read-only `sql` MCP (`sql.list_tables` → `sql.describe`), emits a `SELECT` via `sql.query` (verb + table allowlist enforced by `sqlglot`; covers `paper_content`, `papers`, `chunks`, `chat_sessions`, `messages`, `runs`, `tool_calls`), self-repairs once on error → answer in the user's language + the executed SQL shown as a code block. "Which sessions reference paper X" is answered by joining `papers` ⨝ `paper_content` on `paper_content_id` and filtering on `paper_content.arxiv_id` — `paper_content` is the unique paper identity, `papers` is the per-session membership row (§III-7). **Two-layer scoping (v2.17 fix):** "my library" / "all my papers" questions target `paper_content` (the deduplicated cross-session paper store); "papers in this chat" / "references in this session" questions join `papers ⨝ paper_content WHERE papers.session_id = :session_id`. The SQL planner is seeded with the real schemas of both tables (titles/abstracts/arxiv_id live in `paper_content`; `papers` carries only `session_id`, `paper_content_id`, `enabled`, `added_at`) so it does not hallucinate non-existent columns. **Read-and-act (v2.16):** when the ask is "find the papers I already have on flow matching" the agent shortlists matching `paper_content` rows as candidates (`paper_id="library:<id>"`), auto-attaches up to 2 (`finalize:true` → `add_paper_to_session_dispatch`, always a cache-hit), and emits the same `search_results` SSE event UC-1 uses — so the `SearchResultList` renders auto-added badges + Add buttons for the rest. The agent never writes to `paper_content`/`papers` directly; auto-attach + the user's Add click are the only mutation paths (FR-08).

### UC-7 — Remember, recall, and edit facts (session + global memory)
*"Remember that I'm comparing MoE-routing papers for a survey"* → Router picks `memory` → the memory node first runs the **Memory Gate**: the rule-based safety check accepts this benign content, so the LLM scope classifier runs → writes a `scope='session'` memory (project-scope: a fact tied to this chat/project context) bound to this chat. *"Always answer me in Traditional Chinese"* → gate accepts → `scope='global'` memory (user-scope: a cross-session personal preference). The two scope values embody the functional "user vs project" distinction: `session` == project scope, `global` == user scope. *"Remember to use sk-abc123 as my API key"* → gate **refuses** (sensitive: API key pattern) and returns a visible reason; the `tool_calls` row is `status='rejected'`; the memory is not stored — even though the user explicitly asked. *"Use FastAPI for the backend"* → on add, an LLM conflict-detection call compares against active same-scope memories; if "use Flask" is active, the new memory is saved with `supersedes=<old-id>` and the Flask memory flips to `status='superseded'` + `superseded_by=<new-id>`. On a later turn (even in a different chat for global memories), the `paper_qa` finalizer and the `library_stats` agent receive a **recalled-memory context block** (top-k by FTS over `memories.content WHERE status='active'`, on by default) so answers respect remembered preferences/context. Superseded memories are never injected. *"Update my note about the survey — it's now about expert collapse"* → the memory node recalls the matching memory and `memory.edit`s it (ownership-checked: only this session's memories or global ones are editable, else `rejected`). Agents may also **autonomously** persist a salient durable fact mid-turn via `memory.add` (capped ≤2/turn; gate always runs first). All memory ops live in the dedicated write-capable `memory` MCP (§III-6); the read-only `sql` MCP cannot see the `memories` table. **Memory Manager panel (v2.17, FR-11):** the user opens the Memory Manager from the Composer or sidebar to view all memories — grouped by scope, with active/superseded badges and supersede-chain links. Per-row controls allow editing content, deleting, or toggling active↔superseded (re-activating a superseded note or soft-deactivating an active one). These UI-driven mutations go through the REST endpoints (`PATCH /memories/{id}`, `DELETE /memories/{id}`) — not through the `memory` MCP — because curation is a deterministic human gesture, not an agent action.

### UC-6 — Compare two models on the same question
Presenter toggles **Compare** in the composer, picks two models (e.g. `gemini-2.5-flash` vs `gemini-2.5-pro`), sends the same prompt → two answers stream side-by-side, each with its own routing badge + trace panel. The same enabled-reference set scopes both runs.

The agent's only ingestion-related signal is the `finalize` flag on its shortlisted candidates; the chat endpoint enforces the cap (2) and is the actual caller of the dispatcher. ## I-8. 驗收標準 (Acceptance Criteria)

4. **Trace.** Every chat turn produces a `run_id`; replay-from-SQLite reconstructs the full step list.
5. **Model comparison.** The Compare view renders two model outputs side-by-side for the same prompt, each with its own routing badge + trace. A hand-graded markdown table (10 prompts × 2 models) lives in `eval/results/`.
6. **No silent failure.** A scripted error case (e.g. the LLM provider returns 500, or the Chroma index is missing) shows a visible error in chat AND a red row in the Trace panel — verified once during demo dry-run.

# Part 2 — 技術選型分析 (Technology Selection Analysis)

## II-1. 哪些地方「不能只用 LLM」？

Three places where the LLM is the wrong tool and rules / deterministic code must own the contract:

1. **Citation chunk resolution.** When the user clicks an inline citation, the canvas must scroll to the exact `chunks.id` row in SQLite. The LLM cannot invent that ID — the retriever's output supplies it, then the React renderer rewrites the citation marker into a canvas action keyed on the real chunk ID. Asking the LLM to "remember which chunk it cited" would yield plausible-but-wrong IDs and the canvas would scroll to the wrong text.
2. **Tool-call audit log.** Every entry in `tool_calls` (latency, token counts, status, args) is a measurement, not an opinion. The tracer is rule-based: stopwatch around the call, `try / except` around the return, regex over `args_redacted`. If the LLM judged whether a step "succeeded", the audit log would lose its evidentiary value the moment the LLM disagreed with reality.
3. **MCP scope enforcement.** The rejection of a filesystem `..` traversal, a `DROP TABLE`, or a query against a non-allowlisted table is a hard security boundary. An LLM "scope checker" would have non-zero false-negative rate; rules (regex / `Path.resolve()` / `sqlglot.parse_one`) guarantee 100 % rejection on declared scope.

## II-2. 哪些地方應該用「規則」反而比較好？

| Task | Rule | Why it beats an LLM |
| --- | --- | --- |
| arXiv ID validation | regex `^\d{4}\.\d{4,5}(v\d+)?$` | 100 % accurate, microseconds, zero cost. |
| SQL safety check (NL2SQL output) | `sqlglot.parse_one(sql)` + verb allowlist `{SELECT, WITH}` + table-name allowlist | Deterministic AST inspection — LLM "is this safe?" has non-zero failure rate. |
| Filesystem `..` rejection | `Path.resolve().is_relative_to(workspace_root)` | One canonical answer per path; no probabilistic judgement. |
| API-key redaction in `tool_calls` | regex over `sk-(ant\|proj)-…` and `AIza…` | LLM redactor would miss novel key shapes and is expensive. |
| Routing confidence threshold | numeric `confidence < 0.6 → fallback_to_user=true` | Repeatable, not stochastic. Easy to A/B. |
| Citation chunk lookup | `SELECT … FROM chunks WHERE id = ?` | One canonical answer per ID. |
| Intro2GenAI-hw1 `<think>` block disclosure | DOM regex over `<think>…</think>` | Already works in the reference repo; no model needed. |

**Principle.** Any task with a single deterministically-judgeable correct answer that must be auditable is owned by rules. LLMs are reserved for the semantic / generative work — drafting summaries, routing classification, NL2SQL translation — where multiple acceptable outputs exist and creativity matters more than exactness.

## II-3. 如果不用 RAG，會發生什麼問題？

Five concrete failure modes — each one tied to a specific FR that would break:

1. **Hallucination on the user's library (breaks FR-02 / UC-2).** The user's freshly-indexed papers are not in any LLM's training data. Without RAG, "compare these two papers on expert collapse" returns a plausible-sounding answer composed from the field's *average* paper — and cites section numbers that don't exist.
2. **Citation Canvas dies (breaks FR-03).** The headline UX feature exists only because the retriever attaches a real `chunks.id` to every passage. Without RAG there are no chunk IDs to scroll to; clicking a citation does nothing.
3. **Context-window blowup on multi-paper Q&A (breaks NFR-01 / NFR-05).** Two full 30-page papers ≈ 50 000 input tokens. Stuffing them into every Q&A prompt blows past Gemini Flash's effective practical input ceiling, costs USD ¢×N per turn, and pushes first-token latency well past the 5 s budget.
4. **Knowledge cutoff (breaks UC-1).** A paper published last month doesn't exist in any model's parametric memory. Without RAG over the user's freshly-indexed library, the system cannot answer about anything new.
5. **Personalisation lost (breaks the value proposition).** PaperHub's edge over generic ChatGPT is "the papers *I* have read". Without RAG, the system has nothing personal to retrieve from — it degrades to a generic chat wrapper, and the citation-rate acceptance criterion (I-8 #3) cannot be met.

**Summary.** RAG is not an optional optimisation — it is the architectural decision that lets the Citation Canvas, the source-cited answer, and the personalised library experience all work. The actual RAG pipeline (chunking, embedding, retrieval) is the **in-repo Paper Pipeline** (§III-5.1); the Research Agent threads the retrieved chunk IDs through to the UI as `[chunk:<id>]` markers so the Citation Canvas can resolve every click against a real `chunks` row.

## III-3. ② Orchestrator / Workflow — FastAPI + LangGraph

A single `POST /chat` SSE endpoint. Builds a LangGraph state machine per request, routes via the **Router Agent**, dispatches to one of the sub-agents — **Research Agent / SQL Agent / Report Agent / Memory node** — or the `chitchat` fallback; emits SSE events `routing_decision`, `tool_step`, `token`, `citation`, `final`, `error`.

The agents below are first-class LangGraph nodes — discrete components with their own state, prompts, and tool palette (the Router classifies; the Research / SQL / Report agents + the Memory node handle their intents):

| Agent | Role | Dependencies |
| --- | --- | --- |
| **Router Agent** | Classifies intent → `paper_qa` / `library_stats` / `slides` / `paper_search` / `paper_suggest` / `memory` (fallback `chitchat`; `clarify` short-circuit); picks model tier; resolves anaphora into `resolved_query` + detects `response_language` (v2.11/v2.13); emits the `routing_decision` event | LLM (small tier) |
| **Research Agent** | Handles `paper_search` as a **four-stage LangGraph subgraph** (v2.7 — decomposed from the v2.4-v2.6 single-LLM-turn loop; the read-only contract + shortlist + finalize-marker semantics are preserved, only the internal topology changed). Stages: **(1) `ps_parse`** — small-model LLM splits the user message into N `ParsedRequest`s (deterministic arxiv-ID / DOI pre-scan short-circuits common cases). **(2) `ps_process`** — fan-out per `ParsedRequest` via `asyncio.gather`; each branch runs a bounded Discover→Resolve inner loop (`MAX_REFINEMENT_LOOPS=1`). The **Discoverer** has only `paperhub.search_web` (a structured-output wrapper around `web.*` that makes free-text quoted queries structurally impossible — fields are `paper_hint: str` + `extra_terms: list[str]`, never `query: str`); the **Resolver** has only `papers.search_semantic_scholar` and calls it **exactly once** per request (architectural, not a prompt rule). When SS misses but the Discoverer surfaced an arxiv URL, Python synthesises a `ResolvedPaper` from the extracted `arxiv_id` rather than declaring NotFound. **(3) `ps_finalize`** — Python builds `SearchCandidate`s from `ResolvedPaper`s and emits a `SearchResultsYield` deterministically (the LLM never authors `json:candidates`; block emission is structurally guaranteed). The chat layer applies the finalize cap (max 2) and auto-attaches via `add_paper_to_session_dispatch`. **(4) `Synthesizer`** — short prose summary only, ~30-line prompt. Tool-step streaming is preserved end-to-end: each node drains `drain_tool_calls_since` between stages and emits `tool_step` custom events so the frontend trace panel updates per-stage rather than batched at end. The mission stays *"extend the session's reference set on the user's behalf, informed by what's already in it"* — the prompt context block of currently-enabled references is now consumed by `ps_parse` for disambiguation. Three tools (all read; same JSON-schemas as v2.4-v2.6) form the base palette: `search_library(query, max_results)` (search the user's deduplicated `paper_content` library, excluding papers already in this session), `search_semantic_scholar(query, max_results)` (Semantic Scholar `/graph/v1/paper/search`, primary external discovery), `find_related_papers(paper_id, mode, max_results)` (Semantic Scholar citation graph). **v2.5** — when the MCP registry has a reachable `web` server (`Aas-ee/open-webSearch`, optional external dependency — §III-6), two **discovery-only** tools join the palette: `web.search(query, limit, engines)` (no-key multi-engine general web search; DDG / Bing / Brave / Startpage / Baidu / Sogou) and `web.fetch(url, maxChars)` (fetch + readable extraction of a result page). They surface candidate paper *titles* from blog posts / READMEs / "papers I love" lists when the user's query is short / vague / names a paper indirectly; results carry no `paper_id` and cannot enter `json:candidates` directly — the agent must round-trip a discovered title through `search_semantic_scholar` (or `search_library`) to produce a citable pick. The existing 3-call external-search cap covers `search_semantic_scholar` + `web.search` + `web.fetch` **combined**. With the daemon down or the server unconfigured, the palette reverts to the three v2.4 tools and the prompt registry loads `paper_search/v1` instead of `paper_search/v2`. **No write tool in v2.4** — `add_paper_to_session` was removed from the LLM palette. The agent expresses commitment via `finalize: true` markers in its shortlist; the chat endpoint auto-attaches finalized picks (max 2) before emitting `search_results`. `search_arxiv` is also not in the LLM-visible palette; it survives only as an internal Python helper inside the arXiv-source download path. Per invocation the agent reads `AgentState.history + user_message + current-references context block` (titles, arxiv_ids, abstracts of all currently-enabled session references) and decides whether to (a) emit a clarifying question and stop, (b) `search_library` and shortlist library hits, or (c) `search_semantic_scholar` up to N=3 times with refined queries and shortlist external hits. The agent's final assistant message names the 3-5 picks with reasons + finalize flags; the chat endpoint emits a `search_results` SSE event with the structured candidate list ({paper_id, title, authors, year, abstract, arxiv_id, has_open_pdf, reason, finalize, auto_added, papers_id, error}) just before `final`. The frontend renders these as a `SearchResultList`: finalize-flagged rows show "Added by agent ✓"; the rest show Add buttons the user clicks to ingest. **Three sources of an attached paper**: agent-finalized auto-attach, user clicks Add on a suggested card, user attaches from Library Browser. The agent does not propose papers already present in the context block. Each tool invocation is wrapped by the Tool-Call Tracer (FR-09). The final assistant message names which papers were auto-added (now enabled in the session) and which remaining results the user can attach manually via **Add as reference** buttons on the `SearchResultList` rows. **The agent never toggles `enabled`, never deletes `papers` rows, never mutates `paper_content` / `chunks` directly — see FR-07.** Toggle / remove / manual-attach-from-library are user gestures backed by REST endpoints, used to adjust what the agent extended. Cross-turn refinement is automatic — the next invocation sees the prior conversation and the current curated reference set. Also handles `paper_qa` via the **v2.10 agentic hierarchical pipeline** (§III-5.2): one per-paper subagent per enabled reference navigates that paper's section TOC (`list_sections` / `read_section`, small-tier model) and cites chunks; a flagship finalizer composes the answer over the raw cited chunks. Surfaces `[chunk:<id>]` markers so the Citation Canvas can resolve them. | Paper Pipeline (§III-5), Chroma, LLM (flagship finalizer + small-tier per-paper subagent) |
| **SQL Agent** | Handles `library_stats` (v2.16 — replaces the Plan A `_stub_library_stats` node). Introspection-driven NL2SQL over the read-only `sql` MCP: a small-model call uses `sql.list_tables` + `sql.describe(table)` to learn the schema, emits a `SELECT`, runs it via `sql.query` (which `sqlglot`-validates: verb allowlist `{SELECT, WITH}` + table allowlist, out-of-scope → `status='rejected'`), and **self-repairs once** on error/empty (re-prompt with the error + schema). A flagship call then phrases the answer in `response_language` with the executed SQL embedded as a code block. **Read-and-act:** matching `paper_content` rows can be shortlisted as `library:<id>` candidates and auto-attached (≤2, `finalize:true`) via the same `search_results` SSE path the Research Agent uses (UC-5, FR-08). Also carries the autonomous `memory.add` / `memory.recall` tools and receives the recalled-memory context block (FR-10, active-only). **Two-layer scoping (v2.17):** the planner prompt distinguishes "library" = `paper_content` (all papers the user has ever indexed, deduplicated) from "this chat's references" = `papers ⨝ paper_content WHERE papers.session_id = :session_id`; both table schemas (columns + types) are injected up front so the LLM does not hallucinate columns. Slots `sql_planner/v1` + `sql_repair/v1`. | `sql` MCP (read), `memory` MCP, LLM (small planner + flagship phrasing) |
| **Memory node** | Handles the `memory` intent (v2.16): a small-model loop that recalls candidate memories (`memory.recall`), decides the op (add / edit / forget) + scope (session / global), and writes via the `memory` MCP (ownership-checked; violations → `status='rejected'`). Distinct from the autonomous `memory.add` calls other agents make mid-turn. **Memory Gate (v2.17):** every add path — explicit and autonomous — runs through `agents/memory_gate.py` (deterministic rule check) BEFORE any write; sensitive or dangerous content is refused with a visible reason. **Conflict-supersede (v2.17):** on add, a small LLM call checks active same-scope memories for contradictions; conflicting old memory is flipped to `status='superseded'` and the new memory records `supersedes=<old-id>`. | `memory` MCP, LLM (small) |
| **Report Agent** | Handles `slides`. Reads the session's enabled reference sources, runs the in-repo Slide Pipeline (§III-5.3) — structure planning → per-section generation → emit modern-framework deck (Marp / Slidev / Beamer-via-LangGraph; final framework deferred). Reuses LaTeX helpers copied from `paper2slides-plus` only if the chosen target is Beamer-via-LangGraph. | Slide Pipeline (§III-5.3), Chroma, LLM (flagship) |

`AgentState` is a `TypedDict` carrying `run_id`, `branch` (`''` outside Compare mode; `'A'` / `'B'` inside it — FR-04), `session_id`, `user_message`, `routing_decision`, `enabled_paper_content_ids` (resolved at the start of the turn from `SELECT paper_content_id FROM papers WHERE session_id = :s AND enabled = TRUE` — the membership → content lookup of FR-08), `retrieved_chunks`, `sql_result`, `recalled_memories` (v2.16 — top-k **active** session+global memories injected into `paper_qa` / `library_stats`; v2.17: only `status='active'` rows are ever included), `report_artifact_path`, `final_response`. Each agent is one graph node; the Router's output is a conditional edge. The **Tool-Call Tracer** is wrapped around every model call, MCP call, and pipeline step and writes a `tool_calls` row before returning.

**Framework choice rationale.** Two candidate frameworks were evaluated: **LangGraph** and **Google ADK**. LangGraph was picked because: (a) its node-graph metaphor makes the tool-selection logic visually unambiguous; (b) richer MCP-adjacent Python ecosystem; (c) LangGraph state diffs are diff-friendly in the audit log. Google ADK remains a drop-in alternative — the adapter boundary is the `LlmAdapter` Protocol, swap-out is one module.

## III-4. ③ LLM Module

LiteLLM-based provider adapter — uniform `generate(prompt, model_tier, response_model, slot)` interface; routes to Gemini, Anthropic, OpenAI, or local Ollama based on env config. Prompts live in a YAML registry with versioned slots (`router/v1`, `research_qa/v1`, `sql_planner/v1`, `sql_repair/v1`). Per-call cost is estimated via `litellm.completion_cost` and surfaced in the trace.

Two model tiers:
- **small** — Router intent classification, NL2SQL planner. Latency-sensitive.
- **flagship** — Research Agent answer generation, SQL repair, model-comparison sweep.

## III-5. ④ Knowledge Pipelines (Paper + Slide) & RAG

### III-5.2 paper_qa retrieval (chunks → answer)

**v2.10 — agentic hierarchical retrieval (replaces the v2.2-v2.9 dense-RAG map-reduce).** The Research Agent's `paper_qa` flow is a four-node LangGraph subgraph: `pq_resolve → {pq_empty | pq_dispatch} → pq_finalize`.

1. **pq_resolve** — `enabled_paper_content_ids = SELECT paper_content_id FROM papers WHERE session_id = :session_id AND enabled = TRUE`. Empty → `pq_empty` returns the "add a reference first" message.
2. **pq_dispatch** — fan-out one **per-paper subagent** per enabled paper via `asyncio.gather`. Each subagent (small-tier model, `PAPERHUB_PAPER_QA_SUBAGENT_MODEL`) runs a bounded tool-calling loop with two tools:
   - `list_sections()` — returns the paper's section TOC from `paper_content.sections_json` (`{name, token_count, chunk_count}` per section). Free; no budget cost.
   - `read_section(name)` — returns every chunk in that section, each headed `[chunk:<id>]`. Counts against `MAX_SECTION_READS=5`.
   The subagent navigates the paper **by its section structure** (not by a global cosine-similarity top-k), picks the chunks that answer the question, and cites them with `[chunk:<id>]` markers in a short summary. Python extracts the cited IDs (`chunk:<digits>` token match, dedup) into a `PerPaperPicks`. If the LLM bails without citing, the best-effort fallback hands every chunk the subagent read to the finalizer.
3. **pq_finalize** — the flagship finalizer (`paper_qa_model`) reads the **raw cited chunk text** from every per-paper subagent (NOT a summary of it) plus each subagent's one-line rationale, and composes the user-facing answer, preserving `[chunk:<id>]` markers for the Citation Canvas. All-empty picks → "checked every enabled reference but none relevant" short-circuit (no finalizer call).

**Why agentic over dense top-k:** academic papers carry explicit section structure (`Method`, `Experiments`, `Data`, …) that a global embedding search flattens. Letting a per-paper LLM read the TOC and choose sections matches the question to the right region far better than cosine similarity on chunk vectors, and giving the flagship finalizer the *raw chunks* (rather than an analyst's lossy prose summary) is the key correctness lever. The dense embedder + cross-encoder reranker (`BAAI/bge-reranker-base`, hosted in `paperhub-modelserver` per §III-6) remain in the codebase — `paper_search` discovery and the `slides` pipeline still use them — but `paper_qa` no longer routes through them.

**Observability (harness eval).** Each subagent writes ONE `paper_qa:subagent` `tool_calls` row whose `result_summary_json` records the full loop state: `chunks_read_ids`, `chunks_cited_ids`, `listed_sections`, per-turn LLM content + tool calls (`llm_turns`), per-tool args + chunk IDs returned (`tool_call_log`), and the final summary text. `pq_finalize` writes a `paper_qa:finalize` row. The whole flow is reconstruct-able from `SELECT * FROM tool_calls WHERE run_id = ?` alone — no instrumentation script needed (CLAUDE.md agent-flow observability policy).

**Migration.** Existing `paper_content` rows ingested before v2.10 carry `sections_json = NULL` + chunks from the broken pre-v2.10 chunker; `paperhub-reingest` re-chunks + re-embeds them (preserving `paper_content.id`) so the subagent's `list_sections` has a usable TOC.

### III-5.3 Slide Pipeline (chunks → deck)

The Report Agent's `slides` flow:

1. Resolve `enabled_paper_content_ids` (same scope rule as Q&A — §III-5.2 step 1).
2. **Structure planning** — LLM (flagship tier) reads paper abstracts + section list of each enabled reference (joined from `paper_content`) and produces a `SlidePlan(sections: list[Section])` Pydantic object.
3. **Per-section generation** — LangGraph fan-out: one LLM call per planned section, each retrieving its top-k supporting chunks via the same RAG retriever.
4. **Render** — choose one of three target frameworks. **Decision deliberately deferred** (v2.1): each option has a materially different toolchain footprint and the call should be made when slide-pipeline implementation starts, not at SRS time. The choice cascades into (a) the slide artefact extension under `workspace/chat_session/<session_id>/` (§III-7 currently shows the LaTeX-shaped `slide.tex` / `slide.pdf` — provisional until this resolves), and (b) whether the copied `paper2slides-plus` LaTeX helpers are load-bearing or dead weight. Options:
   - **Marp** (Markdown → PDF / HTML / PPTX via the `marp-cli`; no LaTeX install required; fastest path)
   - **Slidev** (Markdown + Vue components → HTML / PDF; richer interactivity)
   - **Beamer-via-LangGraph** (rebuild `paper2slides-plus`'s LaTeX feedback loop in a clean LangGraph node graph — supports math + figures losslessly, requires a LaTeX install)
4a. **Figure-path resolution across the cache boundary (v2.2).** Each contributing paper's figures live under `workspace/papers_cache/{arxiv/<arxiv_id>,upload/<sha256>}/source/`, not under the per-session output dir. The slide source therefore cannot use bare relative paths like `figures/fig1.pdf` — it must reach across the cache boundary. The Slide Pipeline owns this rewrite at emit time, framework-specific:
   - **Beamer-via-LangGraph** — emit a single `\graphicspath{}` directive at the top of `slide.tex` listing every contributing paper's cache source dir, e.g. `\graphicspath{ {<abs_workspace>/papers_cache/arxiv/2403.01234/source/} {<abs_workspace>/papers_cache/arxiv/2401.05678/source/} }`. Then a per-figure `\includegraphics{fig_name}` resolves by LaTeX's own search order. The pipeline reuses the `\includegraphics` path-rewrite helper copied from `paper2slides-plus/src/latex_utils.py`, adapted to write **absolute** cache paths rather than per-session relative ones.
   - **Marp** — rewrite `![alt](figures/foo.pdf)` to `![alt](file://<abs_workspace>/papers_cache/<key>/source/figures/foo.pdf)` at emit time; `marp-cli` resolves `file://` URIs for local renders.
   - **Slidev** — same as Marp.
   - **Path basis.** Absolute paths (rooted at the workspace) are written into the emitted slide source so the file renders correctly without environment assumptions. Slide source is intentionally **not** portable across machines under v2.2 — moving a workspace requires re-emitting the slide; this is acceptable for the local-only demo scope.
   - **Trace.** This rewrite emits one `tool_calls` row per contributing paper (`slide_pipeline.figure_path_rewrite`) so the audit log shows which cache dirs the deck depends on. If a referenced figure is missing from the cache (e.g. e-print archive omitted it), the row goes `status='error'` and the deck still renders without that figure, with a red row in the Trace panel (NFR-02).
5. **Emit** — output written under `workspace/reports/<run_id>/`, returned as a downloadable chip in the chat.

Reused from `paper2slides-plus`: LaTeX helpers (preamble templates, `\includegraphics` path rewrites, citation-key handling). **Not** reused: the slide-generation orchestrator (`paper2slides-plus/src/core.py:generate_slides` chain — a modern LangGraph rebuild gives us tracing per step and conditional retries by section).

### III-5.4 Vector store

**Chroma** (local file backend, under workspace). Houses every chunk emitted by the Paper Pipeline — keyed by `paper_content_id` (v2.2 two-layer split, §III-7), so each unique paper has exactly one set of vectors regardless of how many sessions reference it. Queried by both `paper_qa` retrieval and `slides` per-section retrieval. Metadata filters include `paper_content_id`, `section`, `char_start`, `char_end` so the Citation Canvas can resolve clicks without extra DB hits.

Eight tables in one SQLite file. Papers are **session-scoped** (each `papers` row carries a `session_id`) — this mirrors the on-disk tree below where every imported paper lives under the chat session that imported it.

```sql
chat_sessions  (id, created_at, title)
paper_content  (id, content_key UNIQUE, kind, arxiv_id NULL, sha256 NULL,
                title, authors_json, year, abstract,
                source_path, source_dir_path, html_path,
                sections_json NULL,   -- v2.10: section TOC for paper_qa subagent
                ingested_at)
papers         (id, session_id → chat_sessions.id,
                paper_content_id → paper_content.id,
                enabled, added_at,
                UNIQUE (session_id, paper_content_id))
chunks         (id, paper_content_id → paper_content.id,
                section, char_start, char_end, text)
messages       (id, session_id → chat_sessions.id, role, content, run_id, created_at)
runs           (id, session_id, routing_decision_json,
                started_at, finished_at, status)
tool_calls     (run_id → runs.id, branch DEFAULT '', step_index, parent_step,
                agent, tool, model,
                args_redacted_json, result_summary_json, latency_ms,
                token_in, token_out, status, error,
                PRIMARY KEY (run_id, branch, step_index))
memories       (id, scope CHECK (scope IN ('session','global')),   -- v2.16 (FR-10)
                session_id NULL → chat_sessions.id,                -- NULL ⇔ scope='global'
                content, created_at, updated_at,
                status TEXT DEFAULT 'active'                       -- v2.17: 'active'|'superseded'
                    CHECK (status IN ('active','superseded')),
                supersedes INTEGER NULL,                           -- v2.17: id this memory replaced
                superseded_by INTEGER NULL)                        -- v2.17: id that replaced this
```

**Two-layer design (v2.2)** — `paper_content` is the *physical* layer: one row per unique paper (keyed by `content_key`, where `arxiv` papers use `arxiv:<arxiv_id>` and uploads use `sha256:<hex>`), owning the on-disk artefacts and **all** `chunks` + Chroma vectors. `papers` is the *logical* / session-membership layer: one row per (session, paper) pair, carrying only the toggle (`enabled`) and `added_at`. Re-importing a paper into another session inserts a new `papers` row pointing at the existing `paper_content_id` — no re-download, no re-extraction, no re-chunking, no re-embedding (UC-2, §III-5.1).

- `papers.session_id` + `papers.enabled` together drive FR-08 (session-scoped reference sources with toggle).
- `paper_content.content_key` is the cache lookup key: `kind='arxiv'` → `content_key='arxiv:<arxiv_id>'`; `kind∈{'pdf_upload','latex_upload'}` → `content_key='sha256:<hex>'`. Exactly one of `arxiv_id` / `sha256` is populated per row (CHECK constraint).
- `chunks` hang off `paper_content_id`, NOT `papers.id` — chunks are content-layer, deduped across sessions.
- `chat_sessions` + `messages` back the chat history.
- `runs` + `tool_calls` back the audit trail (FR-09).
- `memories` (v2.16 + v2.17) backs FR-10 and FR-11 — session-scoped (`session_id` set) or global (`session_id` NULL) facts. `scope='session'` is project-scope (a fact tied to this chat/project context, e.g. "this project uses FastAPI"); `scope='global'` is user-scope (a personal, cross-session preference, e.g. "always answer in Traditional Chinese"). These two enum values ARE the functional user/project distinction — no rename needed. `status` (`active`/`superseded`) + `supersedes`/`superseded_by` ids track the conflict-supersede lifecycle (v2.17); only `status='active'` rows are recalled by agents or injected as context. The `memory` MCP is the sole write surface for agents; `api/memories.py` REST endpoints are the curation surface for the UI (FR-11). Owned exclusively by the `memory` MCP for agent use (the read-only `sql` MCP's table allowlist deliberately excludes it). FTS5 over `content` powers `memory.recall`. The v2.17 status/supersedes/superseded_by columns are added via an idempotent column-add migration.
- `paper_content.html_path` + `chunks.text` back the Citation Canvas (FR-03): the canvas loads the HTML and text-searches it for the chunk's stored `text` (v2.13 — `char_start`/`char_end` index the extracted text, which only aligns with the `<pre>` fallback renders, not the pandoc DOM, so resolution is by content). `char_start`/`char_end` remain useful for RAG provenance + ordering, not for canvas scrolling.
- `tool_calls.branch` discriminates the two halves of a Compare-mode turn under a single `run_id` (FR-04). Empty string for normal turns; `'A'` or `'B'` inside Compare. PK is `(run_id, branch, step_index)` so step indices may legitimately repeat across branches.
- Cache GC policy: **never** (v2.2). `paper_content` rows + their on-disk cache dirs grow monotonically; orphaned content (no referencing `papers` row) is left in place. The single demo workload is small (≤ tens of papers, < 1 GB).
- No `projects`, no `tags`, no `citations` graph, no `session_references` join — drop on sight if anything else accumulates here without an FR backing it. (`memories` is the one v2.16 addition, and it has FR-10 backing it — note-like tables are otherwise still unwelcome.) Re-importing the same `arxiv_id` in a different session creates a new `papers` row but reuses the existing `paper_content` row (no duplicate physical artefacts).

**On-disk layout (v2.2)** — split between a **shared content cache** (`papers_cache/`, keyed by `content_key`) and **per-session output** (`chat_session/<session_id>/`, slides only). Paper sources no longer live under the session tree.

```
workspace/
├── paperhub.db                       # SQLite — all eight tables
├── chroma/                           # Chroma chunk-vector index (per paper_content.id)
├── papers_cache/                     # shared physical layer — one entry per paper_content row
│   ├── arxiv/
│   │   └── <arxiv_id>/               # for paper_content rows with kind='arxiv'
│   │       ├── source/               # unpacked e-print archive (figures + bib + .tex)
│   │       │   └── main.tex          # paper_content.source_path
│   │       ├── source.flattened.tex  # for RAG chunking
│   │       └── source.html           # paper_content.html_path (FR-03)
│   └── upload/
│       └── <sha256>/                 # for paper_content rows with kind∈{pdf_upload,latex_upload}
│           ├── source/               # original archive (LaTeX upload) or source.pdf (PDF upload)
│           ├── source.flattened.tex  # LaTeX uploads only
│           └── source.html           # paper_content.html_path
└── chat_session/
    └── <session_id>/                 # one per chat_sessions.id — slides only, no paper sources
        ├── slide.{tex,md}            # multi-paper Slide Pipeline source (extension depends on framework choice — §III-5.3)
        └── slide.{pdf,html,pptx}     # rendered deck for the session (extension depends on framework choice — §III-5.3)
```

The Slide Pipeline produces **one** source-and-render pair per session (e.g. `slide.tex` + `slide.pdf`, `slide.md` + `slide.html`, …) that references any subset of the enabled papers in that session — exactly the multi-source slide-gen contract from UC-4. Re-running slides overwrites the pair; older runs survive under `workspace/reports/<run_id>/` for trace replay. **Figure-path resolution across the cache boundary is specified in §III-5.3 step 4a** — the slide source must reach figures that now live in `papers_cache/...`, not in a session-local tree.

**Datastore choices:**
- **SQLite** — sole relational datastore (eight tables). (DuckDB was evaluated as an opt-in OLAP path and dropped in v2.16 — the demo fixture set is small enough that plain SQLite handles every `library_stats` query, so a second engine was dead scope.)
- **Chroma** — file-backed under `workspace/chroma/`.
- **Local FS** — split between `workspace/papers_cache/{arxiv,upload}/<key>/` (shared per `paper_content` row) and `workspace/chat_session/<session_id>/` (session-scoped slide output only) per the tree above.

## III-8. Reference-repo integration contract

**Single posture: direct-copy with provenance.** Neither reference project is run as a service. Useful code is copied into the new tree and edited as needed.

- **What to copy.** Utility functions, prompt templates, regex helpers, React components, CSS, MCP-tool wrappers — anything where a service boundary is overkill.
- **What NOT to copy verbatim.** Top-level orchestrators (`paper2slides-plus/src/core.py:generate_slides`, the `chat-with-paper` route, the Node backend of `Intro2GenAI-hw1`) — these are rebuilt in the new LangGraph / FastAPI / React shape.
- **Provenance rule.** Every copied file preserves the original LICENSE / copyright header at the top and adds a single comment: `# Adapted from reference/<repo>/<path> @ <commit>`. Greppable; a future reader can always find the source.
- **`paper2slides-plus` reuse, concretely.** Copy + adapt: `arxiv_utils.py` (arXiv search + e-print download), `pdf_utils.py` (PDF text extraction), `latex_utils.py` (preamble strip, section split, `\includegraphics` rewrite), `file_utils.py` (workspace helpers), individual prompts from `prompts/config.yaml` where they apply. **Skip:** the chat-with-paper orchestrator (can't expose chunk-level citation IDs for the Canvas), `core.py:generate_slides` chain (rebuilt as a LangGraph node graph for per-step tracing).
- **`Intro2GenAI-hw1` reuse, concretely.** Copy + adapt: SSE parsing loop, thinking-block disclosure CSS + DOM logic, history-sidebar layout, fork-from-message UX, MCP indicator chip styles. Re-implement in React (the source is vanilla JS, but the layout grammar and CSS variables port cleanly). **Skip:** the Node + Express backend; PaperHub's FastAPI orchestrator replaces it.

## III-9. Closing principle

When in doubt: **add UX polish before adding backend logic.** When tempted to add a new backend feature, check whether a utility in `reference/paper2slides-plus/` already does it (copy + adapt). When tempted to add a new agent, check whether a single MCP tool would suffice. When tempted to add a new schema table, push back hard.
