# PaperHub

**A UX-first chat shell with its own paper knowledge base + RAG + slide pipeline, assembled by adapting utility functions from two reference projects.**

| Field | Value |
| --- | --- |
| Version | v2.35.0 |
| Date | June 2026 |
| Posture | **UX-first + decomposed reuse.** PaperHub owns its data layer, RAG pipeline, and slide pipeline so every step is traceable. Utility functions are *copied and adapted* from two reference projects rather than consumed as services. |
| Reference projects (utility source) | [`paper2slides-plus`](https://github.com/whats2000/paper2slides-plus) — extraction utils for arXiv / PDF / LaTeX projects + LaTeX helpers, **decomposed; not run as a service** · [`Intro2GenAI-hw1`](https://github.com/whats2000/Intro2GenAI-hw1) — chat-shell + MCP UX idioms ported to React |

# Part 1 — 需求規格書 (Software Requirements Specification)

## I-1. 專案背景 (Project Background)

Two reference projects exist in the author's portfolio that PaperHub draws from:

- **`paper2slides-plus`** — a FastAPI service that turns arXiv IDs / PDFs / LaTeX-zips into Beamer slide decks, with a chat-with-paper Q&A mode. **Decomposed, not run as a service.** Its extraction utilities (arXiv source download, PDF text extraction, LaTeX project handling) are copied into PaperHub and adapted; its chat-with-paper module is *not* reused because it cannot surface chunk-level citation IDs for the Citation Canvas; its slide-pipeline orchestrator is *not* reused because a modern-framework rewrite is needed (§III-6).
- **`Intro2GenAI-hw1`** — a chat UI (Node + Express + vanilla JS) that streams via SSE, manages history, forks conversations, renders thinking blocks, and integrates MCP tools. Its UX idioms are ported to React; its Node backend is not deployed.

PaperHub is therefore a self-contained system: a chat surface, a router, four agents, an in-repo paper knowledge base with RAG, an in-repo slide pipeline, and a tool-call audit log. The four product dimensions PaperHub commits to are problem classification, tool selection, model answer-quality comparison, and traceability — see §I-5 (functional requirements) and §I-8 (acceptance criteria) for the concrete shape of each.

**Engineering principle.** Invest in UX; reuse code by *copying and adapting*, not by service integration. The chat shell, citation canvas, routing badge, trace panel, and LangGraph router are new code; the paper-extraction utilities are copied + edited from `paper2slides-plus`; the chat-shell layout idioms are copied + ported from `Intro2GenAI-hw1`. Every step of the pipeline is in-repo so every trace row points at code we control.

## I-2. 問題描述 (Problem Description)

Today's chat interfaces to LLMs hide three things this product makes first-class:

1. **Opaque tool / model selection.** A general-purpose chat can answer a paper question, run an SQL aggregation, or call an external tool — but the user cannot tell whether the agent picked the right capability or whether a wrong answer was caused by the model, the prompt, or a wrong tool firing.
2. **Ungrounded citations.** When an LLM says *"according to §3.2"*, there is no path from the cited token back to the actual paragraph in the source paper. For research use this is fatal.
3. **Non-reproducible agent runs.** Without a structured per-step audit trail (model, tool, arguments, latency, status), multi-agent answers cannot be debugged or compared across model choices.

PaperHub addresses all three **at the UI surface**: a **routing badge** makes (1) visible per turn; a **citation canvas** with click-to-scroll-and-highlight makes (2) inspectable; a **trace panel** sourced from a `tool_calls` SQLite table makes (3) replayable. The backend doing the actual research work is reused from existing services — the contribution is the inspection layer on top.

## I-4. 使用情境 (Use Cases — 6)

The reading flow is intentionally split into three explicit moments: **search**, **curate**, **discuss / generate**. The user always sees a stable shortlist of "reference sources" attached to the current chat session, and can toggle individual ones on / off — every downstream Q&A or slide generation honours that toggle.

### UC-1 — Search arXiv and shortlist candidates
*"Find recent papers on Mixture-of-Experts routing."* → Router picks `paper_search` → Research Agent runs a **read-only tool-calling loop** (v2.4 — read-only palette, shortlist + optional finalize markers): per turn it decides whether to (a) ask a clarifying question when the user intent is too vague to search (e.g. *"find me good ML papers"* → *"What problem are you trying to solve — routing, training stability, or something else?"*), (b) call `search_library(query)` against the user's deduplicated indexed library, (c) call `search_semantic_scholar(query, max_results)` against `/graph/v1/paper/search` (v2.4 — was `search_arxiv` in v2.3), or (d) call `search_semantic_scholar` again with a refined query (capped at N=3 calls per turn) before shortlisting. The agent's final assistant message names a **shortlist of 3-5 best candidates** with a one-line reason per pick + an optional `finalize: true` marker on the most-confident 1-2 picks. The chat endpoint **auto-attaches finalize-flagged candidates** (cap 2 enforced server-side; calls `add_paper_to_session_dispatch` → triggers download or cache-hit), then emits a `search_results` SSE event with the structured candidate list including `finalize`/`auto_added`/`papers_id`/`error` fields. The frontend renders each as a card with title / abstract / authors / year: auto-added rows show "Added by agent ✓" with toggle/remove controls; suggested-only rows show an **Add as reference** button. **Suggested-only papers are not downloaded** — only finalized (agent-auto) + user-clicked (Add card) + Library-Browser-attached papers trigger ingestion; cache-hit if already indexed. Multi-turn refinement across user replies is supported by the same agent reading prior `AgentState.history`; the user can answer the clarifying question in the next chat turn and the agent picks up the refined intent.

### UC-2 — Promote a paper to a session reference (download-if-needed or add-from-library)
The user clicks **Add as reference** on one or more search results → the Paper Pipeline (§III-5.1) runs. The pipeline's **first step is a cache lookup** on `content_key` (e.g. `arxiv:2403.01234`): if a matching `paper_content` row already exists (from a previous session), the pipeline **skips** download, extraction, HTML render, and chunking entirely, and just inserts a new `papers (session_id, paper_content_id, enabled=true, added_at)` row pointing at the existing content. On cache miss, the pipeline downloads, extracts, renders HTML, chunks, persists a `paper_content` row + `chunks` **once**, then inserts the `papers` row. Either way the paper appears in the **Reference Sources** sidebar panel under the current chat session, marked enabled by default. The user can toggle any reference on / off; toggling does *not* delete anything, it only flips `papers.enabled` for that row.

The user can also bypass the search step entirely by browsing their **already-indexed library** (the union of `paper_content` rows across all of their prior sessions) and attaching any of those papers to the current session — this is the cache-hit path surfaced as a first-class action (v2.3). The backend exposes `GET /papers/library` (paginated list of `paper_content` rows not yet attached to the current session) and `POST /papers/from-library` (inserts a fresh `papers (session_id, paper_content_id, enabled=true, added_at)` row referencing an existing `paper_content_id`; idempotent on `UNIQUE (session_id, paper_content_id)`). Outcome is identical to UC-2's cache-hit path — no download, no re-extraction, no re-chunking.

**Ingestion source selection** (v2.4 — runs through `add_paper_to_session_dispatch(paper_id)` regardless of the trigger): three entry points share the dispatcher — (1) the chat endpoint auto-attaches finalize-flagged candidates from the agent's shortlist (max 2 per turn), (2) `POST /papers` (user clicks Add on a SearchResultList card), (3) `POST /papers/from-library` (user attaches from the Library Browser drawer; always cache-hit by definition). For `paper_id="ss:<paperId>"` the dispatcher fetches the paper's metadata and chooses the download path: (a) if `externalIds.ArXiv` is present, recurse into the arXiv-source ingestion path (LaTeX source is the highest-fidelity ingest — better chunking, math intact, figures resolvable); (b) else if `openAccessPdf.url` is present, fetch the PDF via httpx and run the PyMuPDF extraction path with `kind='pdf_upload'` and `sha256(<pdf-bytes>)` as the `content_key`; (c) else raise `NoIngestibleSourceError` — chat endpoint marks the candidate as `auto_added=false, error="no_ingestible_source"` so the UI can grey out the Add button; `POST /papers` returns HTTP 422 with the same fields. The arXiv-source path is **always preferred over the PDF path** when both are available because LaTeX preserves structure that PDF text extraction loses. The canonical `content_key` (`arxiv:<id>` if arXiv-backed, `sha256:<hex>` for PDF-only) means the same paper picked via any entry point cache-hits future invocations. **Suggested-only candidates (no finalize flag, no user click) are not downloaded** — they remain metadata in the SSE event payload only.

### UC-3 — Multi-paper Q&A with the Citation Canvas *(headline)*
*"How do these two papers differ on expert collapse?"* → Router picks `paper_qa` → Research Agent runs the **v2.10 agentic hierarchical pipeline** (§III-5.2) over the **currently-enabled references**: one per-paper subagent per reference navigates that paper's section table-of-contents (`list_sections` → `read_section`), picks the chunks bearing on the question, and cites them; a flagship finalizer reads the raw cited chunks across papers and generates an answer with inline `[chunk:<id>]` markers tied to real `chunks.id` rows → clicking one opens the **Citation Canvas** on the right, scrolls to the chunk, highlights it. Disabling a reference and re-running the same question dispatches a subagent only for the remaining enabled set.

### UC-4 — Slide generation, editing, and presentation from the enabled reference set
*"Make slides comparing these references."* → Router picks `slides` → Report Agent reads the same enabled set → runs the in-repo Slide Pipeline (§III-5.3): **structure planning** (one thematic cross-paper **synthesis/survey** outline) → **per-section LLM generation** (RAG-grounded, chunks pulled across the enabled papers) → assemble + **compile to a Beamer PDF** (v2.18: framework locked to Beamer-via-LangGraph for conference-ready output — lossless math, real figures, tikz) with a compile-fix retry loop → **speaker notes** (one per page) → returns a **deck chip** in the chat. A single enabled paper degrades to a clean paper-summary deck. The pipeline reuses LaTeX helpers copied from `paper2slides-plus` but is orchestrated by a traced LangGraph subgraph. Toggling a reference off before generation excludes it from the deck.

**v2.21 — generation produces slides only; notes + edits are decoupled, opt-in follow-ups.** *"Make slides comparing these references"* generates **slides without speaker notes**, and the finalize message **hints** the next moves: generate notes · edit a slide · edit the deck. The user can specify **length** (*"a 20-minute talk"*, *"15 slides"*; default 20 min ≈ 15 slides). Notes are authored only when asked (*"幫我加講稿"*), and their **language is independent of the slides'** — *"把講稿變成繁體中文"* re-languages only the notes (the deck stays in its original language) **without regenerating the slides** (the v2.21 bug fix).

The deck is **session-scoped and persistent** — the agent knows whether a working deck exists (`decks` table, one per session). A follow-up `slides` turn is classified by `sl_resolve` (the deck-command classifier) into `generate_notes` / `edit_notes` / `edit_slides` / `regenerate`; an `edit_slides` turn **resolves scope** to the **slide currently on screen** (`current_view_page`), a **named page**, or the **whole deck** — *"make this equation bigger"* while viewing slide 5 edits frame 5; *"add a limitations slide"* edits the deck. Edits are **diff-based** (rewrite only the targeted `deck_slides` frame/note; never full regen) and snapshot a **version** the user can revert to (history parity with `paper2slides-plus`).

The user opens the deck in a **Slides panel** (filmstrip + current slide via react-pdf + collapsible speaker note, FR-12). **Presentation mode** keeps the chat + panel mounted (adding presenter controls — timer, next-slide preview) and opens a **separate fullscreen, slide-only window** on the second screen; every page switch syncs to it via `BroadcastChannel`. Mid-talk, an audience question typed into the composer runs a normal `paper_qa` turn (Citation Canvas answers with citations) while the presentation window holds its page and the panel returns to where it was — the full *present → get asked → answer from the papers → resume* loop.

**v2.25 — F4.5: monolithic tool-using slide agent + version-history UX + configurable preamble (supersedes v2.21's fan-out implementation).** The same chat-driven generation, notes, edit, and presentation surface from v2.21 stays — but the engine underneath is replaced. Generation runs through ONE tool-using slide agent (§III-5.3 v2.25) whose tools are diff edits (`replace_frame` / `insert_frame_after` / `delete_frame` / `replace_preamble`), not a fan-out of per-slide renders. Three deterministic contracts gate `done()`: no hallucinated figures, math-content frames render math (not paraphrased prose), and layout aspect matches figure aspect (advisory). Geometry-grounded overflow detection replaces hardcoded bullet/word rules — text budgets are computed from real text-region geometry per layout (`slide_canvas_budget.yaml`), with figure footprints derived from probed figure dimensions, so a portrait figure on the left side gets a tall-narrow text budget while a landscape figure on top gets a short-wide one. *"Add a slide about the limitations"* / *"split slide 5 — the table doesn't fit"* / *"make this figure bigger"* all flow through the same agent with the existing deck as starting state.

**Configurable preamble.** Default is `paper2slides-plus`'s verbatim default. *"Make slides dark with sans-serif"* — even with no current deck — invokes the slide agent's `replace_preamble(persist=True)` and writes the new style to the new `slide_style_overrides` table; future generations in this session inherit it. *"Remember this style for all future chats"* projects to a global `slide_style_global` memory row. *"Reset the slide style"* is a deterministic router-level command — drops the session override row, no LLM. **No more `decks.theme` enum** (dropped); the preamble is the source of truth.

**Version-history UX (closes Round-2 must-fix #4).** Every successful generation + `done()` snapshots the `(tex, speaker_notes)` to `edit_history/version_<ts>.json` — the mechanism has existed since v2.18; v2.25 activates the UX. The Slides panel renders a **version list**: the currently active version's card renders exactly like the existing "Open Slide" card (same affordances — the user's eye lands on it as the working deck); older versions show a **"Switch to this version"** button → restores `(tex, notes)`, recompiles, updates `decks.current_version_id`, emits a `deck` SSE event. **All subsequent chat revisions then operate on the restored version** — a follow-up *"now make it shorter"* edits forward from that snapshot, not from the latest one.

### UC-5 — Library-stats NL2SQL (read-and-act)
*"How many papers did I add this week? Which sessions reference paper X?"* → Router picks `library_stats` → SQL Agent introspects the schema via the read-only `sql` MCP (`sql.list_tables` → `sql.describe`), emits a `SELECT` via `sql.query` (verb + table allowlist enforced by `sqlglot`; covers `paper_content`, `papers`, `chunks`, `chat_sessions`, `messages`, `runs`, `tool_calls`), self-repairs once on error → answer in the user's language + the executed SQL shown as a code block. "Which sessions reference paper X" is answered by joining `papers` ⨝ `paper_content` on `paper_content_id` and filtering on `paper_content.arxiv_id` — `paper_content` is the unique paper identity, `papers` is the per-session membership row (§III-7). **Two-layer scoping (v2.17 fix):** "my library" / "all my papers" questions target `paper_content` (the deduplicated cross-session paper store); "papers in this chat" / "references in this session" questions join `papers ⨝ paper_content WHERE papers.session_id = :session_id`. The SQL planner is seeded with the real schemas of both tables (titles/abstracts/arxiv_id live in `paper_content`; `papers` carries only `session_id`, `paper_content_id`, `enabled`, `added_at`) so it does not hallucinate non-existent columns. **Read-and-act (v2.16):** when the ask is "find the papers I already have on flow matching" the agent shortlists matching `paper_content` rows as candidates (`paper_id="library:<id>"`), auto-attaches up to 2 (`finalize:true` → `add_paper_to_session_dispatch`, always a cache-hit), and emits the same `search_results` SSE event UC-1 uses — so the `SearchResultList` renders auto-added badges + Add buttons for the rest. The agent never writes to `paper_content`/`papers` directly; auto-attach + the user's Add click are the only mutation paths (FR-08).

### UC-7 — Remember, recall, and edit facts (session + global memory)
*"Remember that I'm comparing MoE-routing papers for a survey"* → Router picks `memory` → the memory node first runs the **Memory Gate**: the rule-based safety check accepts this benign content, so the LLM scope classifier runs → writes a `scope='session'` memory (project-scope: a fact tied to this chat/project context) bound to this chat. *"Always answer me in Traditional Chinese"* → gate accepts → `scope='global'` memory (user-scope: a cross-session personal preference). The two scope values embody the functional "user vs project" distinction: `session` == project scope, `global` == user scope. *"Remember to use sk-abc123 as my API key"* → gate **refuses** (sensitive: API key pattern) and returns a visible reason; the `tool_calls` row is `status='rejected'`; the memory is not stored — even though the user explicitly asked. *"Use FastAPI for the backend"* → on add, an LLM conflict-detection call compares against active same-scope memories; if "use Flask" is active, the new memory is saved with `supersedes=<old-id>` and the Flask memory flips to `status='superseded'` + `superseded_by=<new-id>`. On a later turn (even in a different chat for global memories), the `paper_qa` finalizer and the `library_stats` agent receive a **recalled-memory context block** (top-k by FTS over `memories.content WHERE status='active'`, on by default) so answers respect remembered preferences/context. Superseded memories are never injected. *"Update my note about the survey — it's now about expert collapse"* → the memory node recalls the matching memory and `memory.edit`s it (ownership-checked: only this session's memories or global ones are editable, else `rejected`). Agents may also **autonomously** persist a salient durable fact mid-turn via `memory.add` (capped ≤2/turn; gate always runs first). All memory ops live in the dedicated write-capable `memory` MCP (§III-6); the read-only `sql` MCP cannot see the `memories` table. **Memory Manager panel (v2.17, FR-11):** the user opens the Memory Manager from the Composer or sidebar to view all memories — grouped by scope, with active/superseded badges and supersede-chain links. Per-row controls allow editing content, deleting, or toggling active↔superseded (re-activating a superseded note or soft-deactivating an active one). These UI-driven mutations go through the REST endpoints (`PATCH /memories/{id}`, `DELETE /memories/{id}`) — not through the `memory` MCP — because curation is a deterministic human gesture, not an agent action.

### UC-6 — Compare two models on the same question
Presenter toggles **Compare** in the composer, picks two models (e.g. `gemini-2.5-flash` vs `gemini-2.5-pro`), sends the same prompt → two answers stream side-by-side, each with its own routing badge + trace panel. The same enabled-reference set scopes both runs.

## I-8. 驗收標準 (Acceptance Criteria)

Tied 1:1 to the four product dimensions from §I-1 plus the Citation Canvas headline.

2. **Citation Canvas.** For ≥ 5 distinct inline citations in the demo deck, clicking opens the canvas, scrolls to the chunk, and highlights it within ≤ 200 ms on the demo laptop.

5. **Model comparison.** The Compare view renders two model outputs side-by-side for the same prompt, each with its own routing badge + trace. A hand-graded markdown table (10 prompts × 2 models) lives in `eval/results/`.
6. **No silent failure.** A scripted error case (e.g. the LLM provider returns 500, or a paper's rendered HTML is missing) shows a visible error in chat AND a red row in the Trace panel — verified once during demo dry-run.

# Part 2 — 技術選型分析 (Technology Selection Analysis)

## II-1. 哪些地方「不能只用 LLM」？

Three places where the LLM is the wrong tool and rules / deterministic code must own the contract:

1. **Citation chunk resolution.** When the user clicks an inline citation, the canvas must scroll to the exact `chunks.id` row in SQLite. The LLM cannot invent that ID — the `paper_qa` subagent supplies it (the real `chunks.id` it read from SQLite via `read_section`), then the React renderer rewrites the citation marker into a canvas action keyed on the real chunk ID. Asking the LLM to "remember which chunk it cited" would yield plausible-but-wrong IDs and the canvas would scroll to the wrong text.
2. **Tool-call audit log.** Every entry in `tool_calls` (latency, token counts, status, args) is a measurement, not an opinion. The tracer is rule-based: stopwatch around the call, `try / except` around the return, regex over `args_redacted`. If the LLM judged whether a step "succeeded", the audit log would lose its evidentiary value the moment the LLM disagreed with reality.
3. **MCP scope enforcement.** The rejection of a filesystem `..` traversal, a `DROP TABLE`, or a query against a non-allowlisted table is a hard security boundary. An LLM "scope checker" would have non-zero false-negative rate; rules (regex / `Path.resolve()` / `sqlglot.parse_one`) guarantee 100 % rejection on declared scope.

## II-2. 哪些地方應該用「規則」反而比較好？

| Task | Rule | Why it beats an LLM |
| --- | --- | --- |
| arXiv ID validation | regex `^\d{4}\.\d{4,5}(v\d+)?$` | 100 % accurate, microseconds, zero cost. |
| SQL safety check (NL2SQL output) | `sqlglot.parse_one(sql)` + verb allowlist `{SELECT, WITH}` + table-name allowlist | Deterministic AST inspection — LLM "is this safe?" has non-zero failure rate. |
| Filesystem `..` rejection | `Path.resolve().is_relative_to(workspace_root)` | One canonical answer per path; no probabilistic judgement. |
| API-key redaction in `tool_calls` | regex over `sk-(ant\|proj)-…` and `AIza…` | LLM redactor would miss novel key shapes and is expensive. |
| Routing confidence threshold | numeric `confidence < 0.6 → fallback_to_user=true` | Repeatable, not stochastic. Easy to A/B. |
| Citation chunk lookup | `SELECT … FROM chunks WHERE id = ?` | One canonical answer per ID. |
| Intro2GenAI-hw1 `<think>` block disclosure | DOM regex over `<think>…</think>` | Already works in the reference repo; no model needed. |

**Principle.** Any task with a single deterministically-judgeable correct answer that must be auditable is owned by rules. LLMs are reserved for the semantic / generative work — drafting summaries, routing classification, NL2SQL translation — where multiple acceptable outputs exist and creativity matters more than exactness.

## II-3. 如果不用 RAG，會發生什麼問題？

Five concrete failure modes — each one tied to a specific FR that would break:

1. **Hallucination on the user's library (breaks FR-02 / UC-2).** The user's freshly-indexed papers are not in any LLM's training data. Without RAG, "compare these two papers on expert collapse" returns a plausible-sounding answer composed from the field's *average* paper — and cites section numbers that don't exist.
2. **Citation Canvas dies (breaks FR-03).** The headline UX feature exists only because every cited passage carries a real `chunks.id`. The `paper_qa` subagent reads chunks from the SQLite `chunks` table (by section, `read_section`) and cites them by id; the renderer turns each `[chunk:<id>]` into a canvas action. Without a chunked, section-navigable library there are no chunk IDs to scroll to; clicking a citation does nothing.
3. **Context-window blowup on multi-paper Q&A (breaks NFR-01 / NFR-05).** Two full 30-page papers ≈ 50 000 input tokens. Stuffing them into every Q&A prompt blows past Gemini Flash's effective practical input ceiling, costs USD ¢×N per turn, and pushes first-token latency well past the 5 s budget.
4. **Knowledge cutoff (breaks UC-1).** A paper published last month doesn't exist in any model's parametric memory. Without RAG over the user's freshly-indexed library, the system cannot answer about anything new.
5. **Personalisation lost (breaks the value proposition).** PaperHub's edge over generic ChatGPT is "the papers *I* have read". Without a chunked, per-paper-navigable library, the system has nothing personal to retrieve from — it degrades to a generic chat wrapper, and the citation-rate acceptance criterion (I-8 #3) cannot be met.

**Summary.** Retrieval is not an optional optimisation — it is the architectural decision that lets the Citation Canvas, the source-cited answer, and the personalised library experience all work. The retrieval pipeline is the **in-repo Paper Pipeline** (§III-5.1: structured extraction → section-aware chunking → SQLite `chunks`); `paper_qa` navigates each paper **agentically by its section TOC** (§III-5.2, v2.10) and threads the cited chunk IDs to the UI as `[chunk:<id>]` markers so the Citation Canvas resolves every click against a real `chunks` row.

**What "RAG" means here (v2.27).** The *dense-vector* form of RAG — a Chroma embedding index + cross-encoder reranker queried by cosine similarity — was **removed** in v2.27. An audit found `Retriever.retrieve()` had zero callers after the v2.10 agentic redesign: `paper_qa` reads SQLite sections, `paper_search` uses Semantic Scholar, and `slides` reads `PaperAsset` structure — so the embedder, reranker, Chroma store, and the sibling `paperhub-modelserver` process were dead weight (the embedder wrote vectors nothing queried). "Retrieval" in PaperHub now means **structured chunking + agentic SQLite section-navigation**, not vector search. The five failure modes above are about losing *retrieval altogether* (stuffing raw papers into every prompt); they are not an argument for a dense vector index, which the agentic section reader makes unnecessary.

## III-4. ③ LLM Module

LiteLLM-based provider adapter — uniform `generate(prompt, model_tier, response_model, slot)` interface; routes to Gemini, Anthropic, OpenAI, or local Ollama based on env config. Prompts live in a YAML registry with versioned slots (`router/v1`, `research_qa/v1`, `sql_planner/v1`, `sql_repair/v1`). Per-call cost is estimated via `litellm.completion_cost` and surfaced in the trace.

Two model tiers:
- **small** — Router intent classification, NL2SQL planner. Latency-sensitive.
- **flagship** — Research Agent answer generation, SQL repair, model-comparison sweep.

### III-5.2 paper_qa retrieval (chunks → answer)

**v2.10 — agentic hierarchical retrieval (replaces the v2.2-v2.9 dense-RAG map-reduce).** The Research Agent's `paper_qa` flow is a four-node LangGraph subgraph: `pq_resolve → {pq_empty | pq_dispatch} → pq_finalize`.

1. **pq_resolve** — `enabled_paper_content_ids = SELECT paper_content_id FROM papers WHERE session_id = :session_id AND enabled = TRUE`. Empty → `pq_empty` returns the "add a reference first" message.
2. **pq_dispatch** — fan-out one **per-paper subagent** per enabled paper via `asyncio.gather`. Each subagent (small-tier model, `PAPERHUB_PAPER_QA_SUBAGENT_MODEL`) runs a bounded tool-calling loop with two tools:
   - `list_sections()` — returns the paper's section TOC from `paper_content.sections_json` (`{name, token_count, chunk_count}` per section). Free; no budget cost.
   - `read_section(name)` — returns every chunk in that section, each headed `[chunk:<id>]`. Counts against `MAX_SECTION_READS=5`.
   The subagent navigates the paper **by its section structure** (not by a global cosine-similarity top-k), picks the chunks that answer the question, and cites them with `[chunk:<id>]` markers in a short summary. Python extracts the cited IDs (`chunk:<digits>` token match, dedup) into a `PerPaperPicks`. If the LLM bails without citing, the best-effort fallback hands every chunk the subagent read to the finalizer.
3. **pq_finalize** — the flagship finalizer (`paper_qa_model`) reads the **raw cited chunk text** from every per-paper subagent (NOT a summary of it) plus each subagent's one-line rationale, and composes the user-facing answer, preserving `[chunk:<id>]` markers for the Citation Canvas. All-empty picks → "checked every enabled reference but none relevant" short-circuit (no finalizer call).

**Why agentic over dense top-k:** academic papers carry explicit section structure (`Method`, `Experiments`, `Data`, …) that a global embedding search flattens. Letting a per-paper LLM read the TOC and choose sections matches the question to the right region far better than cosine similarity on chunk vectors, and giving the flagship finalizer the *raw chunks* (rather than an analyst's lossy prose summary) is the key correctness lever. **v2.27 removed the dense embedder + cross-encoder reranker + Chroma store entirely:** an audit found `Retriever.retrieve()` had zero callers — `paper_qa` navigates SQLite sections, `paper_search` resolves via Semantic Scholar, and the `slides` pipeline reads `PaperAsset` structure — so the whole dense-vector stack (and the sibling `paperhub-modelserver` process) was dead weight and is gone (§II-3, §III-5.4).

**Observability (harness eval).** Each subagent writes ONE `paper_qa:subagent` `tool_calls` row whose `result_summary_json` records the full loop state: `chunks_read_ids`, `chunks_cited_ids`, `listed_sections`, per-turn LLM content + tool calls (`llm_turns`), per-tool args + chunk IDs returned (`tool_call_log`), and the final summary text. `pq_finalize` writes a `paper_qa:finalize` row. The whole flow is reconstruct-able from `SELECT * FROM tool_calls WHERE run_id = ?` alone — no instrumentation script needed (CLAUDE.md agent-flow observability policy).

**Migration.** Existing `paper_content` rows ingested before v2.10 carry `sections_json = NULL` + chunks from the broken pre-v2.10 chunker; `paperhub-reingest` re-chunks them (preserving `paper_content.id`) so the subagent's `list_sections` has a usable TOC.

### III-5.3 Slide Pipeline (chunks → deck)

> **v2.33 — Plan F6 extends the v2.25 topology below. F6.1 (narrative planning + the PaperDigest/targeted-read gather rework) is SHIPPED; F6.2 (grounding storage + Sources panel) + F6.3 (custom theme) are PLANNED.** Adds a narrative-planning stage + per-slide grounding traceback + custom theme. The v2.25 monolithic drafter is **retained** (it is what makes today's layout perfect); Plan F6 inserts a stage *before* it and adds storage/UI *after* it. The v2.25 active design (next block) remains accurate for everything except where this block overrides it. The shipped F6.1 topology — compress once (cached `PaperDigest`) → structure from the digest → deterministic `read_section` for exact evidence → always-on streaming — is the "New topology" block below; it **supersedes** the originally-sketched `sl_seed` + `request_context`/`gather_context` dispatch loop (the live UX gate showed the heavy per-aim flagship gather was the wrong worker — see the "Why the heavy per-paper gather is ALSO wrong" lesson below).

**Why (the gap v2.25 leaves).** The v2.25 monolith produces layout-perfect decks but **weak cross-paper narrative**: a multi-paper deck reads as concatenated per-paper summaries — no motivating problem, an orphaned setup (e.g. a taxonomy slide nothing later references), abrupt paper-to-paper seams, and a generic conclusion. Root cause: `gather_context` is per-paper, and the drafter's attention is split between narrative and compile-fitting, so **no stage owns the cross-paper throughline**. F4.4 R1 *had* a planner (`sl_plan_deck → DeckOutline`) but paired it with a **per-slide** renderer (`sl_render_slide`) that lost the whole-deck view → terrible layout → nuked in F4.5. Plan F6 re-introduces planning **without** per-slide rendering.

**Why one-shot planning is not enough (lesson from the first F6.1 build).** A first cut ran `gather_context` blindly once (a fixed, shallow per-paper bundle) and then made **one** outline call over those bundles. The live gate exposed the flaw: gathering had **no aim** (it didn't know the outline), so the outline was trapped planning from a thin, generic bundle → an **8-slide deck for 3 survey papers** (≈2 slides/paper) that read as *distinct topic introductions, not a storyline* — shorter and worse than the v2.25 monolith's ~13 slides. Two root causes: (a) the slide/time budget never reached the planner, and (b) gathering and planning never informed each other. The fix is to make planning **drive** gathering, iteratively.

**Why the heavy per-paper gather is ALSO wrong (second lesson, from the live UX gate).** The first build's *worker* was the F4.5 `gather_context` agent — built to read a paper **comprehensively** and emit a whole-paper bundle (summary + 3–6 figures + "≤5 section excerpts, one per major section"). Used as the orchestrator's reader it (a) ran a **blind whole-paper read per paper** up front, and (b) ran a **full flagship read for EVERY dispatched aim** — including empty/duplicate ones — observed live as **8 redundant flagship gathers of ONE paper** (~50 s each), with the frontend **frozen for minutes** (no SSE during the loop + the ~6-min compile). Root cause is a **contract mismatch**: a *"summarize the whole paper"* agent (mis)used as an incremental reader. People don't read a whole paper to plan a talk — they **skim, internalize each part's gist, then re-read the exact detail when writing**. The fix is **context compression + targeted reads + always-on streaming**, and it makes the blind/heavy gather obsolete.

**New topology — compress once (cached), structure from the digest, read targeted for evidence, stream throughout.** (`PaperDigest` supersedes the deterministic `sl_seed`: it is a *richer* seed carrying each section's **insight**, not just its name.)

```
ingest (ONCE per paper, CACHED, SMALL model — a paper artifact like the F2 PaperAsset):
   paper → PaperDigest { abstract; sections:[{name, insight}]; figures:[{key, one_line}];
                         key_equations:[{latex, role}] }
   each section compressed to its 1–2 line INSIGHT → full coverage in a tiny payload; map-reduce
   per section in parallel; reusable by paper_qa / suggest / compare (not just slides).
        ▼
report:sl_outline   (ORCHESTRATOR — owns the slide/time budget. STRUCTURES the outline from the
   │ ⇅               digest's per-section insights: full coverage, ~free, NO per-deck whole-paper read.
   │ └─ read_section(paper_id, name) → DETERMINISTIC DB fetch of just that section's chunks + chunk_ids,
   │                 dispatched ONLY for a slide's EXACT evidence (a number, a figure's detail, an
   │                 equation). No LLM gather agent per read; no blind whole-paper gather. Concept-driven,
   │                 archetype-shaped, SHOW/SAY split. Each read streams a tool_step the instant it lands.
   │   ⇒ DeckOutline + per-slide evidence (the read chunks + chunk_ids)
        ▼
report:slide_agent  (renders the outline 1:1; writes each slide from its evidence, visually clear)
        ▼
sl_emit             (+ persists per-slide grounding_chunk_ids; NO [chunk:N] in tex)
```

**Always-on event streaming (CRITICAL UX — a deck takes minutes; a frozen frontend reads as broken).** The SSE trace MUST keep moving through the entire turn: (a) each targeted `read_section` flushes a `tool_step` the instant it lands (fast DB fetches stream naturally — vs. the old multi-minute silent gather); (b) a coarse **stage-progress event** opens each phase — *"Reading the papers" → "Planning the talk" → "Drafting slides" → "Compiling the deck (~1 min)"*; (c) the **compile** — one long `pdflatex` off the event loop, the worst silent offender — emits an explicit "compiling…" event + heartbeat so it never looks hung. **Contract: no phase may run silently for more than a few seconds.**

**Narrative-pattern guideline (talk archetype).** The outline's *structure* must fit the talk type, so the orchestrator first selects a **`narrative_pattern`** from paper count + the papers' relationship + the user's intent, then follows that archetype's skeleton when planning slides (this is structural, distinct from F6.3's visual `layout_guideline`). The roster is a **starter set, intentionally extensible** (curated like the theme presets); each pattern is a structural skeleton + guidance the orchestrator's prompt encodes. Starters:

- **`single_paper`** → deep research-talk arc: problem/motivation → approach/method → key results (figures/math) → limitations → conclusion.
- **`sequence`** (evolution of a line of work) → progression: each work advances the previous; every `transition_from_prev` states *the gap the next work closed*.
- **`comparison`** (methods/approaches) → shared problem framing → each method assessed on **common axes** → head-to-head trade-offs → when-to-use guidance.
- **`synthesis`** (default for unrelated/parallel multi-paper) → thematic grouping → cross-cutting themes → one fused takeaway.
- **`tutorial`** (teaching-oriented) → prerequisites → core concept → worked example → practice/takeaways.
- **`application`** (case-study / deployment) → real-world problem → solution approach → results in context → lessons.
- **`taxonomy`** (a single survey/review paper) → field framing → walk the taxonomy branches/method families the survey covers → trends/open problems.

**Survey/review papers are internally multi-work.** A survey is NOT one contribution — it organizes many works/methods/themes. So `sl_seed` flags **paper type** (research vs survey/review — inferred from title/abstract cues like "survey"/"review"/"overview" and a section structure of many parallel topic sections), and for a survey the orchestrator treats its **taxonomy branches / surveyed method families / themes as the planning units** (like multiple papers) and dispatches a detail gather per branch — rather than compressing the whole survey into one shallow slide. A single survey → `taxonomy`; a survey mixed with primary papers → its branches join the `synthesis`/`comparison`/`sequence` as first-class threads.

More archetypes can be added without code changes to the loop (they are prompt/skeleton entries). The orchestrator **infers** the pattern (e.g. 1 paper → `single_paper`; "compare …" → `comparison`; "evolution/progress of …" → `sequence`; teaching framing → `tutorial`; else `synthesis`); the user's wording overrides the inference; if none fits, it falls back to a general motivation → development → synthesis arc. The chosen `narrative_pattern` is recorded on the `DeckOutline` + in the trace, so a mis-structured deck is diagnosable.

**North star — model a graduate student making the slides.** The entire loop exists to reproduce how a grad student actually prepares a talk from papers: skim to understand what each paper contains; pick a structure that fits the talk type (a single-paper journal-club talk, a comparison, an evolution story…); **go back into the papers** to pull the specific figure / result / equation each planned slide needs; write each slide from that fetched material; and iterate until the deck is the right depth and tells a coherent story. Every design choice below (deterministic seed, archetype guideline, aimed detail dispatch, outline+context handoff, budget) serves that behavior — a grad student would not ship 8 thin topic-intro slides for three survey papers, and neither should this.

**Concept-driven, not a highlight reel.** A well-formed deck first decides the **concepts it needs to convey** and gives each slide **one concept to develop** (backed by the evidence fetched for it) — it does NOT pick a handful of salient highlights and stitch them together (the precise failure mode of the first build). So the orchestrator plans the *conceptual backbone* first (what ideas, in what order, build the argument for this archetype), then assigns each concept a slide and dispatches detail gathers to support it. `key_message` is the slide's concept; the gathered context is its evidence.

**Slide content form (the right visual per concept).** A concept must be *shown* the right way, so each `OutlineSlide` carries a **`content_form`** the orchestrator chooses from the concept + archetype + the evidence available; the drafter renders that form. The roster is a **starter set, extensible** (prompt/skeleton entries):

- **`problem_statement`** → prose/bullets framing the gap · **`method_walkthrough`** → a pipeline/flow diagram (paper figure or drawn) · **`comparison_table`** → methods × common axes · **`results`** → a table/chart/numbers from REAL gathered data · **`concept_figure`** → a figure illustrating an idea (an existing paper figure, OR an **agent-drawn TikZ/LaTeX schematic** when no paper figure fits) · **`equation_focus`** → a key equation + notation · plus the structural forms **`title`/`agenda`/`section_divider`/`synthesis`**.

**Agent-drawn figures — capability + hard guardrail.** Beamer is lossless for TikZ (a reason it was chosen, SRS v2.18), so the drafter MAY author an original **TikZ schematic** to illustrate a concept the papers don't provide a figure for. **Guardrail (Contract #1 extended):** a drawn figure is a **conceptual schematic ONLY — it must NEVER fabricate data or results.** Anything presenting measured values (charts, results tables, benchmark numbers) must come from a real paper figure or from real numbers in the gathered context; the drafter may not invent a data figure. A drawn schematic illustrates an idea (a pipeline, a relationship, a taxonomy) and reads as a diagram, not as the paper's evidence.

**The loop (orchestrator = `sl_outline`, worker = `gather_context`).** The high-level pass is **`sl_seed` — deterministic, no LLM**: per paper it returns the abstract/contribution, the **section list**, and the **figure inventory**. That map is what makes aiming possible — the orchestrator can only dispatch a detail request against a section/figure that appears in the seed (it cannot ask for what it doesn't know exists). The orchestrator never reads sections itself — it **delegates** detail fetches to the **agentic, task-driven `gather_context` worker** (given an *aim* + paper scope, the worker reads the relevant sections/figures and returns targeted excerpts + the chunk IDs it read). Each round it may dispatch several aims in parallel; it terminates when the outline is structurally complete (motivation → per-thread development → synthesis) and every content slide is grounded, OR the round budget is hit. **The drafter receives the FINAL REPORT = the outline (scope/structure) + the accumulated gathered context** (the real material per slide) — so it writes from fetched content, not from the outline's one-line goals alone. Depth emerges from the loop digging where the narrative is thin; the slide budget gives it a target count — fixing the 8-slide thinness + the invisible storyline. Per the agent-flow record principle, every round (seed map, draft outline, dispatched aims, returned excerpts + chunk IDs) is recorded so the planning is reconstructable from the trace.

**`DeckOutline` schema** (revives R1's `DeckOutline`/`PlannedSlide`, drops the rigid `pattern_kind`):

- `DeckOutline`: `talk_title`; `narrative_pattern` (the chosen archetype — `single_paper`/`sequence`/`comparison`/`synthesis`/…); `audience_intent` (the "evaluate the user's need" output — default = walk through the cited references); `narrative_arc` (the throughline: problem framing → per-section bridges → synthesis takeaway); `slides: [OutlineSlide]`.
- **The slide SHOWS, the notes TELL (load-bearing principle).** Real slide design thinks about the spoken narration alongside the slide: the slide carries only what must be **visually clear** (the concept, a few key points, the figure/table), and **anything that needs *explaining* — words you would *say* — becomes a comment routed to the speaker-note model**, not text on the slide. So every `OutlineSlide` separates **SHOW** (concise on-slide content) from **SAY**: it carries a **`speaker_note_hint`** (the fuller explanation + the `transition_from_prev` bridge) that flows to the notes agent (F6.1b), and the drafter renders ONLY the visual-clear part. A rendered bridge/explanation reads as an awkward secondary title and bloats the slide — the iron rule is *if it's words you'd speak, it does not go on the slide.* The deck's connectedness is delivered by visible **structure** (agenda · section dividers · synthesis) + the spoken **notes**; keeping the explanation off the slide is also the primary over-text mitigation.
- **Over-text detection + refine (concise|split).** A slide that fills to the frame edge (no bottom margin) is over-dense even when it doesn't trigger a hard `Overfull \vbox`. Detect via `body_token_count > text_budget_tokens` (not only Overfull) and refine by EITHER tightening to concise (drafter revise) OR — preferred, and 1:1-safe — the orchestrator **splitting a dense concept into two outline slides at plan time** (it owns the count; one concept ≤~4 bullets per slide; never overload a slide).
- `OutlineSlide`: `slide_index`; `goal` (its job in the arc); `key_message` (**the ONE core concept the slide exists to convey**); `content_form` (how to SHOW it — `problem_statement`/`method_walkthrough`/`comparison_table`/`results`/`concept_figure`/`equation_focus`/`agenda`/`section_divider`/`synthesis`/…); `transition_from_prev` (the connective bridge today's decks lack); `paper_id | null` (null for title/agenda/synthesis slides); `figure_key?` (an EXISTING paper figure, constrained to the gathered inventory — Contract #1 still holds, no hallucinated figures; a `concept_figure` with no `figure_key` signals the drafter may draw a TikZ schematic — never fabricated data); `equation_refs?`; `support_excerpts: [str]` (the gathered evidence the drafter writes this slide from); `grounding_chunk_ids: [int]`.

**Per-slide discipline — one core concept, not a highlight grab-bag.** A well-formed slide *teaches one point*. `key_message` is that single concept; the drafter selects from the gathered context only the material (a figure, a result, an equation, ≤a few supporting bullets) that lands it, and nothing else. The orchestrator must give every content slide a **sharp, distinct** `key_message` — no two slides making the same point, no slide that is just "assorted highlights from §4." This is also the loop's definition of *well-formed*: a slide is done when it has a clear core concept **and** the grounded material to support it; if a planned slide is still a vague bucket of highlights, that is a thread the orchestrator must split (into concept-sized slides) or dispatch more detail for — not emit.

**Authoritative-1:1 contract** (the load-bearing decision). The drafter renders **one frame per `OutlineSlide`** and may tighten content/layout but MUST NOT add, drop, or split slides. This (a) keeps `grounding_chunk_ids` → final PDF page mapping **exact** (by `slide_index` — the traceback UI depends on it), and (b) stops the drafter reverting to its own stitched structure. A planned slide that cannot be made to fit falls back to the existing overflow handling (tighten → degraded-but-shipped), NOT auto-split. The outline is **internal scaffolding** — no user-facing outline-approval gate (the user sees the deck + the Sources panel). Per the agent-flow record principle, `sl_outline.record_result` records the full outline + per-slide chunk IDs, so a misnarrated deck is diagnosable from the trace alone.

**Per-slide grounding traceback** (separate storage; the citation is NEVER on the deck):

- **Schema:** `deck_slides.grounding_chunk_ids_json TEXT` — JSON int array; NULL for synthesis/title slides. `sl_emit` writes it by `slide_index`. No `[chunk:N]` ever enters `frame_tex`.
- **API:** the deck-slides response gains `grounding_chunk_ids` + light resolved metadata (paper title, §section, snippet) via the existing `api/chunks.py` resolver — the same one the Citation Canvas uses.
- **UI — "Sources (this page)":** a strip directly **under the deck-page render** (the currently-empty band above the speaker-notes block), **synced to the on-screen page**. A compact row of clickable chips (`paper §section`, snippet on hover); click → **Citation Canvas** at that chunk (reuses `[chunk:id]` resolution + `CitationMarker`/`CitationCanvas`; inherits the `dom_id=null` text-search fallback). Pages with no grounding show a quiet "synthesis — no single source" line. All strings i18n'd across the four locales.

**Custom theme / layout guideline** (shapes BOTH the outline and the rendering):

- **Input:** a small curated **preset gallery** of common talk styles (starter set: `academic-classic` = today's Berlin/dolphin default · `clean-light` = white bg / black text / minimal · `dark-minimal`; roster intentionally minimal + extensible — the operator curates) PLUS **natural-language** refinement ("…but bigger figures, brand blue") where NL overrides the preset. Presets are BOTH a frontend gallery pick AND agent-selectable by name ("make it an IEEE-style deck").
- **`layout_guideline`** (structured, agent-distilled): `{preset_id, text_density, figure_emphasis, prefers_columns, tone, accent_color, fonts, notes}`. `sl_outline` reads it (e.g. high `figure_emphasis` → more figure-led slides; low `text_density` → tighter messages); it is also mapped into the Beamer preamble (theme / colours / fonts).
- **Persistence:** extend `slide_style_overrides` with a `layout_guideline_json` column alongside `preamble_tex`; precedence UNCHANGED (session override → global memory → default). The drafter's `replace_preamble` tool still serves raw-TeX power users. `GET /slides/presets` serves the gallery.

   1b. **Targeted reads** — the orchestrator structures from the digest (full coverage) and dispatches **deterministic `read_section`** only for a slide's exact evidence. **v2.25 — F4.5 monolithic tool-using slide agent (the active design).** A 3-step traced flow. The middle step is **ONE agent** that owns the whole deck across draft AND revise, using diff-edit tools rather than holistic full-deck regeneration.

```
report:gather_context   (per-paper subagent fan-out)
        │   N × PaperContextBundle (narrative + figure inventory w/ PIL-probed dimensions + equations w/ role)
        ▼
report:slide_agent      (ONE tool-using agent — owns the deck)
   ├─ initial_draft(deck_tex)              first cut, full preamble + every frame
   ├─ compile_check()                      → {ok, page_count, compile_errors,
   │                                         frame_overflow[…], unrendered_math_frames[…]}
   ├─ density_check(tex_excerpt)           same detector, no pdflatex (speculative-split verify)
   ├─ replace_frame(n, tex)                surgical patch of ONE frame — diff edit, not rewrite
   ├─ insert_frame_after(n, tex)           split an overfull frame into two
   ├─ delete_frame(n)
   ├─ replace_preamble(tex, persist=True)  style mutation; persist=True writes slide_style_overrides
   └─ done()                               gated on contracts (see below)
        │   deck.tex
        ▼
sl_emit                 (deterministic: figure-key audit → write decks + deck_slides → snapshot version → SSE)
```

**Stage 1 — `report:gather_context`** (the **only** stage that fans out; fan-out is correct here because each paper's context extraction is independent IO with no design decisions). Per paper, a light agentic subagent uses `list_sections` / `read_section` / `read_figure_block` over F2's `PaperAsset` to assemble ONE `PaperContextBundle`:

- Narrative summary (contribution, method, key results — text)
- Figure inventory: per figure, `{key, width_px, height_px, aspect_ratio (probed via PIL at gather time), caption, role}` — this is the gap that broke F4.4 R1; fixed at the source
- Key equations: per equation, `{latex, role, notation_legend}` — the agent decides at draft time which to render
- ≤5 representative section excerpts the agent may quote

The R1 `PaperTalkBrief` schema is gone; `PaperContextBundle` is a looser narrative-shape bundle the monolithic agent reads holistically.

**Stage 2 — `report:slide_agent`** (the whole creative middle). System prompt mirrors the `paper2slides-plus` content contract (title → executive summary → intro → method [bulk] → results [quantified] → conclusion) but lets the agent decide layout per frame against three deterministic feedback loops. Tool palette listed above. The agent loops draft → compile_check → diff-edit → compile_check until `done()` passes the contract gate. Tool-call budget caps the loop at ~15 calls + a token budget; if exhausted, ships imperfect deck (same fallback as `compile_with_revise`).

**Three hard contracts** (the deterministic guards that prevent F4.4 R1's failure modes from recurring):

| # | Contract | Detector signal | Gate | Recovery |
| --- | --- | --- | --- | --- |
| 1 | **No hallucinated figures.** Every `\includegraphics{key}` in `deck.tex` must reference a key in the staged figure inventory (the deck-unique-keyed `slides/figures/` from F4 step 4a, preserved). | Post-`done()` audit (`verify_and_fix_graphics`, deterministic — survives from F4.4) | Silent deterministic correction at `sl_emit` | Replace unknown key with `\textit{[figure omitted]}`. NEVER prompts the LLM — auditing is the point. |
| 2 | **Math-content frames render math.** A frame whose **title or body topic-keyword-matches** any `key_equation.role` from the gathered bundles MUST contain a math block (`\[...\]` / `equation` / `align` / inline `$...$`). Paraphrasing math into prose is forbidden where the frame's purpose IS the math. Other frames are unaffected — a Motivation slide need not have math; a systems paper may have zero math frames. | `compile_check()` returns `unrendered_math_frames[…]` with `{frame_index, frame_title, matched_equation_role, matched_equation_latex, paper_idx, recommendation: "replace_frame with equation_centered layout"}`. Role-keyword overlap is deterministic (split role on `_`, lowercase, intersect with frame title + body keywords; threshold 0.6 to avoid false positives). | `done()` REJECTED until empty | Agent re-spawns one more turn to call `replace_frame` with the matched equation rendered. |
| 3 | **Layout aspect matches figure aspect.** Portrait figure → left-column or right-column slot; landscape → top-band or bottom-band; square → either. Mismatch (e.g. portrait figure crammed into landscape-top slot) is flagged. | `compile_check()` returns `frame_overflow[i].layout_aspect_mismatch` per frame | **Advisory.** Iterates within tool-call budget. At budget exhaustion, ships with logged warning. | Agent calls `replace_frame` with an aspect-matched layout from `slide_layout_examples.yaml`. |

Compile errors are a fourth gate: `done()` rejected until zero compile errors OR tool-call budget exhausted (then ships imperfect deck per the existing `compile_with_revise` precedent — degraded deck beats lost deck).

**Geometry-grounded overflow detection** replaces F4.4's `"≤4 bullets, ≤12 words"` hardcoded heuristics. The detector reads `slide_canvas_budget.yaml` (operator-tunable — **no thresholds in Python**) which declares per layout pattern:

```yaml
# slide_canvas_budget.yaml — example entries
constants:
  frame_canvas_cm: [12.8, 6.5]                # full canvas after headline/footline at 14pt
  line_height_cm: 0.65
  bullet_pitch_factor: 1.6                    # multiplied with line_height for itemize spacing
  chars_per_cm: {en: 12, cjk: 8}              # 14pt body; script-aware
  chars_per_word: 6

layouts:
  text_only:
    figure_region_cm: [0, 0]
    text_region_cm:  [12.8, 6.5]
    matches_aspect: any                       # no figure → aspect is N/A

  figure_left_half_portrait:                  # column layout for tall figures
    matches_aspect: "0.5..1.3"                # portrait or near-square
    figure_region_cm: [6.0, 6.5]
    text_region_cm:  [6.8, 6.5]
    text_structure_hint: "4-5 stacked bullets, ≤14 words each"

  figure_top_full_landscape:                  # banner layout for wide figures
    matches_aspect: ">=1.5"                   # landscape only
    figure_region_cm: [12.8, 4.0]
    text_region_cm:  [12.8, 2.5]
    text_structure_hint: "1 short takeaway OR ≤3 single-line bullets"

  equation_centered:                          # math frames — used for contract-2 fixes
    figure_region_cm: [0, 0]
    equation_region_cm: [12.8, 3.0]
    text_region_cm:  [12.8, 3.5]
    text_structure_hint: "notation legend + 1-2 sentence interpretation"

  # … plus figure_left_third, figure_inset_text_around, comparison_two_figures_side_by_side,
  #     table_only, title_frame, closer_frame (regular-theme — no [plain] + custom centering)
```

**Token budget per layout** = `(text_region_height ÷ (line_height × bullet_pitch_factor)) × (text_region_width × chars_per_cm) ÷ chars_per_word`. So `figure_left_half_portrait` budgets ≈ `(6.5 ÷ 1.04) × (6.8 × 12) ÷ 6 ≈ 85` Latin tokens (≈ 57 CJK tokens). `figure_top_full_landscape` budgets ≈ `(2.5 ÷ 1.04) × (12.8 × 12) ÷ 6 ≈ 61` Latin tokens — wide-but-short bands hold less than narrow-but-tall columns even at the same area, because bullet pitch dominates. **Same area, different shape, different capacity.**

**Per-frame overflow signal** in `compile_check()`'s return:

```json
{
  "frame_index": 7,
  "frame_title": "Simulation and Real-World Evaluation",
  "page_number": 8,
  "matched_layout": "figure_left_half_portrait",
  "signals": {
    "canvas_total_cm2": 83.2,
    "figure_footprint_cm2": 39.0,
    "remaining_text_region_cm": [6.8, 6.5],
    "text_budget_tokens": 85,
    "body_token_count": 187,
    "overage_tokens": 102,
    "pdflatex_overfull_pt": 23.7,
    "layout_aspect_mismatch": false,
    "exceeds_canvas_budget": true
  },
  "recommendation": "split_frame",
  "split_hint": "figure_to_own_frame_then_text"
}
```

`figure_footprint_cm2` is computed by resolving each `\includegraphics[width=W,height=H,keepaspectratio]` against the figure's actual aspect ratio from the inventory — `keepaspectratio` means the smaller of `(w, w·aspect)` wins, so the math is deterministic, no estimate. Same detector backs `density_check()` (no compile, for speculative splits before paying the `pdflatex` cost).

**Framework (v2.18 — deferred decision resolved, retained for historical context): Beamer-via-LangGraph.** The v2.1 three-way deferral (Marp / Slidev / Beamer) is settled in favour of **Beamer/LaTeX** because the product target is **conference-ready academic slides**: lossless math (incl. each paper's custom `\newcommand` macros), real `\includegraphics` figures from the papers, and tikz/pgfplots when a genuine vector diagram is needed — Markdown frameworks (KaTeX + Mermaid) are toy-grade for a conference talk. The cost is accepted: **`pdflatex` is now a hard dependency** for the `slides` intent (absence → the agent returns a "install a LaTeX distribution" message, not a crash), and the compile-fix retry loop is load-bearing. The copied `paper2slides-plus` LaTeX helpers become load-bearing rather than dead weight. Default Beamer theme `metropolis` with a built-in fallback (`Madrid`) when the theme package/fonts are absent (non-fatal compile warning).

**Agent topology (v2.21 — three decoupled sub-flows; F4 redesign).** A deck is a **living artifact**: generation, speaker-note authoring, and editing are independent, opt-in operations rather than one bundled run. `report_graph` keeps the single `slides` router intent, but a **deck-command classifier** (small LLM, runs only when a `decks` row already exists for the session) emits a `DeckCommand {action ∈ generate_notes | edit_notes | edit_slides | regenerate, target ∈ current | page:N | all, note_language?}` and routes to one of three sub-flows. No existing deck → **GENERATE**.

- **GENERATE** *(slides only — notes opt-in)* — `sl_resolve → sl_understand → sl_narrate → sl_draft (frame-only) → sl_coherence → sl_assemble → sl_verify_figures → sl_compile → sl_emit`. Produces slides with **no speaker notes**; the finalize message **hints** the next moves (generate notes · edit a slide · edit the deck).
- **NOTES** *(opt-in, separate; `generate_notes` / `edit_notes`)* — `nt_load → nt_author (per page, in `note_language` — independent of the slide language) → nt_finalize → nt_persist → emit`. Writes `deck_slides.note_text` + `note_language`; never touches frames.
- **EDIT** *(diff-edit, scoped; `edit_slides` / `edit_notes` with `target`)* — load the target `deck_slides` row(s) → single targeted LLM rewrite of the frame (or note) → `sl_verify_figures → sl_compile` → persist. Never a full regen. `regenerate` is the explicit "redo the whole deck" → GENERATE.

**Length budget (v2.21).** `sl_narrate` takes a real slide budget: default **20 min ≈ 15 slides** (`slides ≈ minutes × 0.75`, clamped 8–30) + `depth ∈ {overview, standard, deep}`; the user's "20-minute talk" / "15 slides" flows in (was emergent from `pdflatex`).

The PhD-grade single-pass core below (v2.19) is the GENERATE path's stage detail. `agents/report_graph.py` is a traced LangGraph subgraph that simulates how a PhD builds a talk, consuming each paper's **`PaperAsset`** (§III-5.1: figure inventory with captions, equations as LaTeX, structured sections). Cost is no object — flagship throughout, multi-pass. GENERATE nodes:

1. **`sl_resolve`** *(small LLM — the deck-command classifier)* — resolve `enabled_paper_content_ids` (Q&A scope rule); load any existing `decks` row; if one exists, classify the turn into a `DeckCommand` (`generate_notes` / `edit_notes` / `edit_slides` / `regenerate`, with `target` from `current_view_page` or a named page, and an optional `note_language`) and route to NOTES / EDIT / GENERATE accordingly; no deck → GENERATE. Also parse any length intent (target minutes / slide count) into the narrate budget. Empty enabled set → "enable a reference first". `pdflatex` absent → install message.
2. **`sl_understand`** *(per paper, flagship fan-out)* — build a structured brief per paper: core contribution, problem, method, key results, **which figures matter (by `PaperAsset` inventory id)**, key equations (LaTeX). Reads `PaperAsset.structure` + section excerpts. This is the "read the paper deeply" step.
3. **`sl_narrate`** *(flagship)* — produce ONE coherent **synthesis** storyline (single paper degrades to a clean summary), sized to the **length budget** (target slide count + depth). Output is a slide-by-slide outline following the `paper2slides-plus` content contract (slide 1 = title / first-author / date / arXiv-url; slide 2 = executive summary; then intro → method [the bulk] → results [quantified] → conclusion): per slide `{title, communicative goal, key points, assigned figure id (from inventory) OR equation (LaTeX), source chunk ids, source paper ids}`. **Figures are assigned here, only from the real inventory** (grounding).
4. **`sl_draft`** *(per slide, flagship fan-out — frame-only, v2.21)* — for each outlined slide, generate a **concise Beamer frame**: ≤~4 bullets (≤~12 words each, quantified claims) + at most the one assigned figure/equation. **Speaker notes are NOT drafted here** (v2.21 decoupling — the NOTES sub-flow authors them on demand). Frames forbid `\pause`/overlays so PDF pages == logical frames.
5. **`sl_coherence`** *(flagship, holistic)* — one pass over ALL drafted frames: remove redundancy, fix flow + transitions, enforce consistent style + density discipline (the reference's `update` stage, smarter).
6. **`sl_assemble`** *(deterministic)* — preamble (`\usetheme{metropolis}` + fallback, `\input{ADDITIONAL.tex}`) + title + frames; merge `ADDITIONAL.tex` macros from each LaTeX-source paper; **stage the assigned figures** into one `slides/figures/` dir under deck-unique keys + a single `\graphicspath` (4a).
7. **`sl_verify_figures`** *(deterministic — HARD no-hallucination guarantee)* — every `\includegraphics{key}` MUST reference a staged inventory key; any unknown → the frame is sent back to a targeted re-draft (`sl_draft` on that frame) or the figure is dropped. **A non-existent figure can never reach the PDF.** Records the verification result.
8. **`sl_compile`** *(LOOP — self-correcting layout)* — `pdflatex`; success requires not just exit-0 but **no Overfull-vbox / off-slide overflow**. On a fatal error OR an Overfull-vbox, a flagship `revise` call rewrites/splits the offending frame(s) with the log → recompile. Iterate (generously bounded — quality over speed). A deck that still won't compile emits `status='error'` + last good PDF (NFR-02).
9. **`sl_emit`** — write/update `decks` + one `deck_slides` row per final frame (`frame_tex`, `page_start`/`page_end`; `note_text` NULL until the NOTES flow runs), snapshot a version, emit the `deck` SSE event + a **hinting** final prose (generate notes · edit a slide · edit the deck). GENERATE emits `has_notes=false`.

**NOTES sub-flow (v2.21 — opt-in; `generate_notes` / `edit_notes`):** `nt_load` (read `deck_slides` + final PDF page map) → `nt_author` (flagship, per page: a **rich** spoken narrative in `note_language` — independent of the slide language; `edit_notes` rewrites existing `note_text` toward the instruction, e.g. re-language to Traditional Chinese) → `nt_finalize` (reconcile to FINAL page count, one note per page, pad/repair gaps) → persist `deck_slides.note_text` + `note_language`, refresh the derived `decks.speaker_notes_json` → emit `has_notes=true`. **Never touches frames** (this is the bug fix: re-languaging notes no longer regenerates slides).

**EDIT sub-flow (v2.21 — diff-edit; `edit_slides` with `target`, detailed in plan F4):** load the target `deck_slides` row(s) by `target` (`current` → `current_view_page`; `page:N`; `all`) → single targeted LLM rewrite of the frame (reuse `beamer_helpers` extract/replace) → splice into the deck `tex` → `sl_verify_figures → sl_compile` → snapshot + update the affected `deck_slides` rows. `regenerate` → GENERATE.

4a. **Figure staging (v2.19 — replaces the v2.18 cross-boundary `\graphicspath` of raw cache dirs).** The v2.18 approach (point `\graphicspath` at each paper's raw cache dir) caused **cross-paper filename collisions** (`img_2` in two papers → wrong image) and Windows-backslash compile failures. v2.19: `sl_assemble` **stages only the assigned figures** from each `PaperAsset` into one `slides/figures/` dir under **deck-unique keys** (e.g. `p{idx}-{stem}`), with a single forward-slashed `\graphicspath{ {.../slides/figures/} }`. `\includegraphics[width=…,height=…,keepaspectratio]{key}` then resolves unambiguously. One `report:figure_stage` `tool_calls` row records the staged key→source map.

**Tracing** (observability policy): `report:resolve`, `report:understand[paper]` (brief + chosen figure/equation ids), `report:narrate` (the outline + per-slide assigned asset ids), `report:draft[slide]` (frame + note + asset id + chunk ids), `report:coherence`, `report:figure_stage`, `report:verify_figures` (unknown-key rejections + redrafts), `report:compile` (**always** records the trimmed log incl. Overfull lines + attempts + page_count — not only on failure), `report:notes_finalize`, `report:emit`. Full reconstruct-from-DB; no rendered prompts.

**Reused from `paper2slides-plus` (copy+adapt, provenance per NFR-03):** `latex_helpers` (defs/packages extraction, `ADDITIONAL.tex`, `sanitize_frametitles`), `compile` (the `compile_with_revise` loop, now Overfull-aware), `beamer_helpers` (frame extract/replace for edits), `history` (`VersionHistory`). **Not reused:** the single-paper orchestrator. **Nuked from F1:** `slides_plan/section/notes` prompts, the `figures.py` flat-stem approach, the `_revise` no-op.

**Persistence + version history.** One **current deck per session** (`decks` table, §III-7, `UNIQUE(session_id)`). Snapshot `(tex, speaker_notes)` bundles under `workspace/chat_session/<sid>/slides/edit_history/version_<ts>.json` after every successful compile; `VersionHistory.list_versions` / `restore_version` (lockstep tex+notes, then recompile to regenerate the PDF). ### III-5.4 Chunk store (SQLite — vector store removed v2.27)

Chunks live in the **SQLite `chunks` table** (content layer, keyed by `paper_content_id` per the v2.2 two-layer split, §III-7), so each unique paper has exactly one set of chunks regardless of how many sessions reference it. Each row carries `id, paper_content_id, section, char_start, char_end, text, dom_id` — everything `paper_qa` (section navigation, §III-5.2) and the Citation Canvas (`GET /chunks/{id}`, FR-03) need, with no extra DB hops. **There is no vector store.** The Chroma embedding index + cross-encoder reranker were **removed in v2.27**: the dense `Retriever.retrieve()` had zero callers after the v2.10 agentic redesign (the embedder wrote vectors nothing queried), so it, the reranker, Chroma, and the sibling `paperhub-modelserver` process were all deleted (see Revision History + §II-3). Re-introducing semantic/dense retrieval (e.g. a future compare-by-similarity feature or semantic memory recall) would re-add an embedding model + a vector index.

## III-7. ⑥ Database — Schema + On-Disk Layout

Ten tables in one SQLite file (v2.25 added `slide_style_overrides`). Papers are **session-scoped** (each `papers` row carries a `session_id`) — this mirrors the on-disk tree below where every imported paper lives under the chat session that imported it.

                sections_json NULL,   -- v2.10: section TOC for paper_qa subagent
                asset_status NULL,    -- v2.20: 'latex'|'pymupdf_only'|'marker_pending'|'marker_ready'|'marker_failed'

                paper_content_id → paper_content.id,

**Two-layer design (v2.2)** — `paper_content` is the *physical* layer: one row per unique paper (keyed by `content_key`, where `arxiv` papers use `arxiv:<arxiv_id>` and uploads use `sha256:<hex>`), owning the on-disk artefacts and **all** `chunks` rows. `papers` is the *logical* / session-membership layer: one row per (session, paper) pair, carrying only the toggle (`enabled`) and `added_at`. Re-importing a paper into another session inserts a new `papers` row pointing at the existing `paper_content_id` — no re-download, no re-extraction, no re-chunking (UC-2, §III-5.1).

- `papers.session_id` + `papers.enabled` together drive FR-08 (session-scoped reference sources with toggle).
- `paper_content.content_key` is the cache lookup key: `kind='arxiv'` → `content_key='arxiv:<arxiv_id>'`; `kind∈{'pdf_upload','latex_upload'}` → `content_key='sha256:<hex>'`. Exactly one of `arxiv_id` / `sha256` is populated per row (CHECK constraint).
- `chunks` hang off `paper_content_id`, NOT `papers.id` — chunks are content-layer, deduped across sessions.
- `chat_sessions` + `messages` back the chat history.
- `runs` + `tool_calls` back the audit trail (FR-09).
- `memories` (v2.16 + v2.17) backs FR-10 and FR-11 — session-scoped (`session_id` set) or global (`session_id` NULL) facts. `scope='session'` is project-scope (a fact tied to this chat/project context, e.g. "this project uses FastAPI"); `scope='global'` is user-scope (a personal, cross-session preference, e.g. "always answer in Traditional Chinese"). These two enum values ARE the functional user/project distinction — no rename needed. `status` (`active`/`superseded`) + `supersedes`/`superseded_by` ids track the conflict-supersede lifecycle (v2.17); only `status='active'` rows are recalled by agents or injected as context. The `memory` MCP is the sole write surface for agents; `api/memories.py` REST endpoints are the curation surface for the UI (FR-11). Owned exclusively by the `memory` MCP for agent use (the read-only `sql` MCP's table allowlist deliberately excludes it). FTS5 over `content` powers `memory.recall`. The v2.17 status/supersedes/superseded_by columns are added via an idempotent column-add migration.
- `decks` (v2.18) backs FR-12 + UC-4 — one **current** deck per session (`UNIQUE(session_id)`). Holds the tex/pdf paths, `speaker_notes_json` (`{page: note}` — **v2.21: a derived cache** of `deck_slides.note_text`, kept for the `deck` SSE `has_notes` flag + the `GET /deck` back-compat shape), the `plan_json` outline (kept for legacy v2.18-22 reasoning; F4.5's monolithic agent does not consume it), `page_count`, `current_version_id` (**v2.25**: which `edit_history/version_<ts>` file the working state was last restored from — `NULL` = latest), and `contributing_paper_ids_json` (which `paper_content` rows the deck draws from). **The `theme` column was dropped in v2.25 / F4.5** — the resolved preamble (session override → global memory → default `slide_style_default.tex`) is the source of truth, no separate enum. Version history is **file-based** (snapshot `(tex, speaker_notes)` bundles under `chat_session/<sid>/slides/edit_history/`, parity with `paper2slides-plus/src/history.py`), not a table — v2.25 activates the **UX** of this mechanism via `current_version_id` + the frontend version-list cards. Added via idempotent migrations (v2.18 created; v2.25 added `current_version_id` + dropped `theme`).
- `deck_slides` (v2.21) backs the decoupled slide/note operations (FR-12) — one row per **final** frame `(deck_id, slide_index)`, holding the frame's LaTeX, its (opt-in, possibly differently-languaged) speaker note, and the PDF page span it occupies. This is the per-slide granularity that lets the NOTES sub-flow author/re-language notes and the EDIT sub-flow diff-edit a single frame **without re-serializing the whole deck** — the fix for "re-languaging the notes regenerated the slides". `decks.speaker_notes_json` is rebuilt from these rows on every notes/edit write. Added via an idempotent migration alongside `decks`.
- `slide_style_overrides` (v2.25, F4.5) backs the configurable Beamer preamble (§III-5.3 v2.25). One optional row per session; when present, its `preamble_tex` replaces the `paper2slides-plus`-derived default for that session's deck generations + edits. `source` records *how* the override landed (user explicitly asked → `user_request`; agent inferred during a normal edit → `agent_inferred`; projected from the user's global `slide_style_global` memory row at session boot → `global_memory_projection`). Resolution priority on every `initial_draft` / `replace_frame` is **per-session override → global memory (`memories` row with `metadata.kind='slide_style_global'`) → default `slide_style_default.tex`**. Mutated by the slide agent's `replace_preamble(persist=True)` for LLM-creative restyling; deterministic command intercepts at the router level (no agent spawn) handle "reset slide style" (`DELETE`) + "remember this style for all future chats" (write `slide_style_global` memory row from this override). Replaces the F4.2 `slide_style_profiles.yaml` multi-profile registry and the dropped `decks.theme` column — a single default + per-session override + optional global memory is the whole surface. Added via an idempotent migration; `decks.theme` is dropped in the same migration (a no-op for sessions that never customised theme since it defaulted to `'metropolis'` and never actually persisted on regenerate per the closed Round-2 must-fix #2).
- `decks.current_version_id` (v2.25, F4.5) names the file id (`version_<ts>` slug) under `workspace/chat_session/<sid>/slides/edit_history/` that the deck's working state was last restored from. `NULL` ⇒ working state is the latest snapshot (the natural state after a fresh generation or compile). When the user clicks "Switch to this version" on an older snapshot in the frontend Slides panel, `POST /sessions/{sid}/deck/versions/{file}/restore` restores `(tex, notes)` lockstep, recompiles, **and** sets `current_version_id=<file>` so the UI can highlight which version card is the active one. All subsequent chat revisions operate on this restored state — the slide agent reads `decks.tex_path` after restore, and a new `done()` snapshots a new file under `edit_history/` with `current_version_id` advancing to that fresh file id. Closes Round-2 must-fix #4 (the underlying file-based snapshot mechanism existed since v2.18; v2.25 is purely the UX activation + the "which one is active" pointer).
- `paper_content.html_path` + `chunks.text` back the Citation Canvas (FR-03): the canvas loads the HTML and text-searches it for the chunk's stored `text` (v2.13 — `char_start`/`char_end` index the extracted text, which only aligns with the `<pre>` fallback renders, not the pandoc DOM, so resolution is by content). `char_start`/`char_end` remain useful for RAG provenance + ordering, not for canvas scrolling.
- `tool_calls.branch` discriminates the two halves of a Compare-mode turn under a single `run_id` (FR-04). Empty string for normal turns; `'A'` or `'B'` inside Compare. PK is `(run_id, branch, step_index)` so step indices may legitimately repeat across branches.
- Cache GC policy: **never** (v2.2). `paper_content` rows + their on-disk cache dirs grow monotonically; orphaned content (no referencing `papers` row) is left in place. The single demo workload is small (≤ tens of papers, < 1 GB).
- No `projects`, no `tags`, no `citations` graph, no `session_references` join — drop on sight if anything else accumulates here without an FR backing it. (`memories` is the one v2.16 addition, and it has FR-10 backing it — note-like tables are otherwise still unwelcome.) Re-importing the same `arxiv_id` in a different session creates a new `papers` row but reuses the existing `paper_content` row (no duplicate physical artefacts).

**On-disk layout (v2.2)** — split between a **shared content cache** (`papers_cache/`, keyed by `content_key`) and **per-session output** (`chat_session/<session_id>/`, slides only). Paper sources no longer live under the session tree.

```
workspace/
├── paperhub.db                       # SQLite — all nine tables
├── papers_cache/                     # shared physical layer — one entry per paper_content row
│   ├── arxiv/
│   │   └── <arxiv_id>/               # for paper_content rows with kind='arxiv'
│   │       ├── source/               # unpacked e-print archive (figures + bib + .tex)
│   │       │   └── main.tex          # paper_content.source_path
│   │       ├── source.flattened.tex  # for RAG chunking
│   │       └── source.html           # paper_content.html_path (FR-03)
│   └── upload/
│       └── <sha256>/                 # for paper_content rows with kind∈{pdf_upload,latex_upload}
│           ├── source/               # original archive (LaTeX upload) or source.pdf (PDF upload)
│           ├── source.flattened.tex  # LaTeX uploads only
│           └── source.html           # paper_content.html_path
└── chat_session/
    └── <session_id>/                 # one per chat_sessions.id — slides only, no paper sources
        └── slides/                   # v2.18 — the session's one current Beamer deck (decks table)
            ├── deck.tex              # decks.tex_path
            ├── deck.pdf              # decks.pdf_path (NULL until first successful compile)
            ├── ADDITIONAL.tex        # merged+deduped macros from all contributing LaTeX-source papers
            ├── speaker_notes.json    # {page: note}

```

The Slide Pipeline produces **one current Beamer deck per session** (`deck.tex` + `deck.pdf`, framework locked v2.18 — §III-5.3) that references any subset of the enabled papers in that session — the multi-paper synthesis contract from UC-4. Editing/recreating overwrites `deck.tex`/`deck.pdf` but **snapshots the prior `(tex, speaker_notes)` into `edit_history/` first** (version-revert parity with `paper2slides-plus`); figures are never copied here — `\graphicspath` reaches into `papers_cache/...` (§III-5.3 step 4a).

**Datastore choices:**
- **SQLite** — sole relational datastore (nine tables). (DuckDB was evaluated as an opt-in OLAP path and dropped in v2.16 — the demo fixture set is small enough that plain SQLite handles every `library_stats` query, so a second engine was dead scope.)
- **Local FS** — split between `workspace/papers_cache/{arxiv,upload}/<key>/` (shared per `paper_content` row) and `workspace/chat_session/<session_id>/` (session-scoped slide output only) per the tree above.

## III-9. Closing principle

When in doubt: **add UX polish before adding backend logic.** When tempted to add a new backend feature, check whether a utility in `reference/paper2slides-plus/` already does it (copy + adapt). When tempted to add a new agent, check whether a single MCP tool would suffice. When tempted to add a new schema table, push back hard.
